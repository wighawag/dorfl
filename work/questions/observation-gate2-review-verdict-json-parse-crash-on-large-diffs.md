## Q1

**What should become of this signal? It reports a real, reproducible Gate-2 robustness bug: `parseReviewVerdict` (packages/dorfl/src/review-verdict.ts:106-119) does a strict `JSON.parse` of the review agent's verdict with no salvage/repair, no per-finding length cap, and no retry, so a malformed verdict on a large diff throws `ReviewParseError` that is NOT caught/routed inside `runGate2Review` (integration-core.ts) and crashes the whole `do` run. The observation proposes three fix directions: (1) make the verdict parse fault-tolerant and route a parse failure to needs-attention with a transient-infra/config-error cause instead of throwing; (2) harden the review agent's output contract (strict minified JSON, escaped control chars, capped per-finding length, fenced-block extraction + lenient repair pass before strict parse); (3) on a parse crash still PUSH the kept work branch to origin and mark the lock stuck so recovery does not need a manual mirror->origin push. Should this be promoted to a task (and if so, which of the three fix directions are in scope), folded into a broader gate-robustness PRD, or dropped? Please answer in your own words.**

> Claim verified against current code on 2026-06-25: review-verdict.ts:106-119 does strict `JSON.parse` then `throw new ReviewParseError(...)` with no salvage/length-cap/retry; the gate call site in integration-core.ts (`runGate2Review`, around L2649-2741) maps a non-approve verdict to `review-blocked`/needs-attention but has no try/catch turning a `ReviewParseError` into a routed outcome, so the throw propagates and strands the run (orphaned active lock, no origin push, no PR). The recurrence is well-attested: at least eight done-task bodies record this exact 'review verdict was not valid JSON' crash on large diffs at deep payload positions (~5.7k-8.1k), each AFTER a fully green Gate-1 build, each recovered by hand. No open task in work/tasks/ready or work/tasks/backlog tracks the fix; the only nearby done task (needs-attention-routing-resilient-honest-requeue-safe) hardens needs-attention ROUTING against network/git outages, which fires AFTER routing and does NOT cover the parse-time throw that happens before any routing. The item is status:open, needsAnswers:true, with no pre-existing `## Open questions` block, so this is the native observation-triage question.

_Suggested default: Promote to a task scoped to fix directions (1) + (2): make `parseReviewVerdict` fault-tolerant so a parse failure is caught at the `runGate2Review` call site and routed to needs-attention with a transient-infra/config-error cause (never an unhandled throw, never a silent approve), and harden the review prompt's output contract (strict minified JSON, escaped control chars, capped per-finding length, fenced extraction + a lenient repair pass before the strict parse). Treat fix direction (3) (push the kept branch + mark the lock stuck on a parse crash) as a recommended follow-on within the same task if the routing already pushes on the needs-attention path, since once the throw is caught the standard needs-attention routing should handle branch push and lock state._
