## Problem Statement

`install-ci`'s composite setup action (`dorfl-setup`, generated by `packages/dorfl/src/install-ci-core.ts:generateSetupAction`) provisions **only what dorfl and the agent harness need**: Node 22 (`actions/setup-node@v4`, hardcoded), the dorfl CLI (`npm install -g dorfl` in `registry` mode, or a pnpm build-from-source in `workspace` mode, which is auto-detected ONLY inside the dorfl monorepo), the `pi` harness, git identity, and AI-provider auth. Every capability workflow is then `checkout → dorfl-setup → dorfl <verb>`.

There is **nothing that provisions the user's OWN project toolchain.** When the agent a `dorfl` verb spawns then runs the project's `verify` command (e.g. `pnpm -r build && pnpm -r test`, `cargo test`, `go test ./...`), it depends on a toolchain nobody installed: no project-pinned pnpm/corepack, no `rustup`, no project Node version, no `pnpm install` of the project's deps, no system packages. The generated CI can build dorfl-shaped repos but not an arbitrary consumer project. `ResolvedCIConfig` carries `authMode` / `providers` / `harness` / `installSource` and **no project-shaped field**, and there is no escape hatch, template, or hook.

(Note on dorfl's OWN Node: dorfl declares `node >=18` — it runs on any reasonably-modern Node and needs no PINNED version. So this is purely a PROJECT-toolchain gap, not a dorfl-runtime-version gap; the fix is to provision the project's toolchain and keep dorfl's runtime from conflicting with it, NOT to add a dorfl-Node-version knob.)

Two adjacent problems compound it:

1. **dorfl's own runtime can conflict with the project's.** The single `setup-node@v4` step pins the runner's GLOBAL Node to 22. A project that wants a different Node (or no Node) inherits dorfl's Node when its build runs, with no way to re-point it. dorfl must be installable + runnable regardless of whether the project uses Node/pnpm at all, AND must not silently become the project's toolchain.
2. **A repo that pins dorfl in its `devDependencies` does NOT get that dorfl in CI.** `registry` mode is always `npm install -g dorfl` (latest), and capability workflows invoke a bare `dorfl` resolving to that global install. There is no detection of a project-local `node_modules/.bin/dorfl` and no version pinning, so CI runs a different (skewed) dorfl than the one the repo declares — a real version-skew foot-gun.

I want the generated CI to be able to build/verify an ARBITRARY project: provision the project's toolchain through a native, provider-appropriate escape hatch; keep dorfl's own runtime isolated from (and non-conflicting with) the project's; and prefer the project's pinned dorfl over a global one.

## Solution

Extend `install-ci` along three layered axes, framed so the global dorfl bootstrap, the project provisioning, and the project-pinned dorfl compose cleanly:

- **(A) Isolate dorfl's runtime; document the boundary.** Keep `setup-node@v4` for dorfl (A1) — dorfl declares `node >=18`, so it runs on whatever Node is present and needs NO pinned-version knob. If a Node-only project provisions its own Node via the (B) hook (which runs first), dorfl simply resolves on that. Document the boundary explicitly: a custom or conflicting project toolchain is supported **only** via (B)'s escape hatch; otherwise it is unsupported (a deliberate, honest line, not a silent gap). install-ci does NOT try to DETECT a likely conflict and warn — the boundary is documentation, not detection logic. A self-contained dorfl binary (no ambient Node) and a private-Node-prefix isolation are recorded as FUTURE options (A2/A3), not built here.

- **(B) A provider-namespaced, opaque, pass-through project-setup hook — the "template/customization thing".** The user supplies the steps that make THEIR project verifiable (corepack/pnpm@pinned, `rustup`, a project Node, `pnpm install`, system packages). The keystone decision: **users write their host CI's NATIVE syntax, and dorfl splices it verbatim into ITS generated composite action** between dorfl-install and agent-run. dorfl does NOT invent a portable step-DSL and does NOT parse the snippet. For GitHub that means native Actions step YAML (`uses: actions/setup-node@v4`, `uses: pnpm/action-setup@v4`, `uses: dtolnay/rust-toolchain`, any marketplace action). The hook is keyed BY PROVIDER (an opaque blob per provider), so the provider-agnostic core stays dumb and each adapter owns its own injection point + native shape — no portability promise we cannot keep, and no stuck state for non-GitHub.

- **(C) Prefer the project-pinned dorfl over the global one (C1 + C3).** Generated invocations resolve a project-local `node_modules/.bin/dorfl` first, and fall back to the global install (C1). Once (B) has run the project's `pnpm install` and dorfl is a devDep, that resolver finds the pinned dorfl (C3). The global install stays as the always-present BOOTSTRAP so a config-less repo still works zero-config.

The layering is the keystone: the **global dorfl bootstraps** (existing behaviour) → **(B) provisions the project** (and may install a pinned dorfl as a devDep) → **(C)'s resolver lets that pinned dorfl take over**. (A) keeps dorfl's runtime from poisoning the project's, and (B) is where any genuinely custom toolchain (including a conflicting pnpm/Node) is expressed.

The non-GitHub trap is resolved by construction: because (B) is provider-namespaced opaque pass-through, a GitLab/Buildkite adapter consumes its OWN native snippet (its `before_script`, its job shape), never a mistranslation of GitHub YAML. The core only delegates "emit my provider's project-setup at the right point" to the active adapter — the same provider-agnostic-core / thin-adapter philosophy the existing `CIProviderContext` seam already uses.

## User Stories

1. As a maintainer of a non-trivial project, I want the generated CI to install MY project's toolchain (pnpm at my pinned version, rust, a specific Node, system packages) before the agent runs my build, so `dorfl verify` actually succeeds in CI instead of failing on a missing toolchain.
2. As a GitHub user, I want to express that project setup in NATIVE GitHub Actions step syntax (`uses:` marketplace actions, `run:` steps), NOT a dorfl-invented mini-format, so I reuse the ecosystem I already know and dorfl never mistranslates it.
3. As a maintainer, I want that project-setup hook to round-trip through `--config` / `--export-config` like every other `install-ci` knob, so my CI is reproducible from config and re-runnable.
4. As a maintainer on a non-GitHub CI provider, I want the project-setup hook to NOT be locked to GitHub Actions syntax — I supply my provider's native snippet — so adopting dorfl on GitLab/Buildkite later is not blocked by a GitHub-shaped abstraction, and dorfl never pretends to portably translate one provider's steps to another.
5. As a maintainer whose project uses NO Node (pure Rust/Go), I want dorfl to still install and run in CI, so dorfl's own Node requirement is dorfl's private concern and does not force Node onto my project.
6. As a maintainer whose project pins a DIFFERENT Node/pnpm version than dorfl uses, I want a documented, supported way to provision my version (the (B) hook) and a clear statement that without it the conflicting case is unsupported, so I am never silently broken — and because dorfl only needs `node >=18`, my (B)-provisioned Node satisfies dorfl too.
7. As a maintainer who pins `dorfl` in my `devDependencies`, I want CI to run THAT pinned dorfl, not a global latest, so CI and my laptop run the same dorfl and a version bump is deliberate, not silent.
8. As a maintainer of a config-less repo, I want dorfl to still work zero-config (a global bootstrap install), so adopting the project-pinned path is an opt-in improvement, never a precondition.
9. As a maintainer, I want the project-setup steps injected into the SHARED composite action (`dorfl-setup`), not duplicated per workflow, so every capability inherits the same provisioning once and there is one place to change it.
10. As a maintainer, I want my project-setup steps to run FIRST in the composite action (before dorfl-install and AI-auth), so my toolchain is in place before anything that might depend on it.
11. As a maintainer, I want the running CI job to STILL never edit `.github/workflows/**` (inherits `runner-in-ci` US #9) — the project-setup hook lands in the generated composite action via the emitter, not by hand-editing a workflow, so no new self-rewrite path opens.
12. As a maintainer, I want dorfl to NOT semantically validate / parse my project-setup snippet (only a light structural sanity check + a generated-output snapshot), so the escape hatch stays a true escape hatch and does not re-introduce the mini-format it exists to avoid.

## Out of Scope

- **A self-contained dorfl binary (A2)** and **a private-Node-prefix isolation (A3).** Recorded as future runtime-isolation options; (A1) + the (B) hook cover the need now without them.
- **A `nodeVersion` knob for dorfl's own Node.** DROPPED: dorfl declares `node >=18` and runs on any modern Node, so pinning its Node is over-engineering; the project's Node is the project's concern (the (B) hook).
- **Conflict DETECTION + warnings.** install-ci does not sniff `.nvmrc` / `packageManager` / `rust-toolchain.toml` to warn about a likely missing (B) hook. The boundary is documented, not enforced by detection.
- **A preset library for (B)** (curated pnpm-via-corepack / node / rust snippets). DEFERRED — the raw native-syntax escape hatch ships first; presets are later sugar over the same hook, not part of this SPEC.
- **Injecting dorfl's needs into the user's OWN existing workflows.** Explicitly rejected (see ADR `install-ci-project-provisioning-native-passthrough`) — it breaks dorfl's ownership of the generated files and the idempotent upgrade contract.
- **A portable, cross-provider step-DSL.** Deliberately NOT built: the hook is provider-namespaced opaque pass-through precisely so dorfl never promises to translate one provider's steps to another.
- **Building the non-GitHub adapter itself.** GitHub is the first (and only, here) adapter; the seam is shaped so a second provider can supply its own native snippet later WITHOUT a core rewrite, but that adapter is its own work.
- **Changing `installSource` semantics** (`registry` / `workspace` auto-detection). Untouched; this SPEC adds the project layer above it, not a change to the dorfl-install path.
- **The `install-ci` wizard's broader surface** (auth modes, provider selection, branch protection, capability selection). Owned by `runner-in-ci` and its existing tasks; this SPEC ADDS the project-provisioning + project-pinned-dorfl knobs to that surface, it does not re-build the surface.

## Further Notes

- **Why provider-namespaced opaque pass-through is the keystone:** it gives GitHub users full native syntax (the headline UX) at near-zero cost (it reuses the existing "interpolate an opaque YAML fragment" pattern in `generateSetupAction`), invents no format to maintain, and structurally cannot get stuck on non-GitHub providers (each adapter consumes its own native snippet). The alternative — a normalized portable step schema — would be the thing that gets stuck, because no honest portable translation of "`uses: dtolnay/rust-toolchain`" to a GitLab `before_script` exists.
- **Why the layering matters:** keeping the global dorfl as an always-present bootstrap (C's fallback) is what lets (B) optionally install a project-pinned dorfl WITHOUT a chicken-and-egg problem (you need a dorfl to run, but the pinned one only exists after the project's `pnpm install`). Bootstrap-global → provision-project → prefer-local is the clean resolution.
- **Prior art to read at tasking time:** `packages/dorfl/src/install-ci-core.ts` (`generateSetupAction`, `buildSetupArtifacts`, the `ResolvedCIConfig` / `CIConfigFile` shapes, the `CIProviderContext` seam, the capability-emitter registry), `install-ci.ts` (the wizard / `--config` / `--export-config` / `installSource` auto-detection), `install-ci-github.ts` (the GitHub adapter + validator style to mirror), `.github/workflows/advance-lifecycle.yml` (the `checkout → dorfl-setup → dorfl <verb>` shape every capability uses), and whitesmith (`~/dev/github/wighawag/whitesmith`, `src/providers/github-ci.ts`) for the original composite-action / `--fake`-snapshot patterns.
- **Relationship to `runner-in-ci`:** this SPEC is an additive layer on the `install-ci` scaffolder `runner-in-ci` owns. It `taskedAfter` `runner-in-ci` because it extends that scaffolder's config + composite action; it does not fork a competing install-ci.
- Task this SPEC with `to-tasks`. The three axes are naturally separate, file-orthogonal tasks under one umbrella: **(A)** the documented isolation boundary (no knob, no detection — a docs/output line + a snapshot asserting dorfl's `setup-node` still emits); **(B)** the provider-namespaced opaque project-setup hook spliced FIRST into the composite action (the keystone, largest); **(C)** the project-local-then-global dorfl resolver. (B) and (C) have a SOFT ordering (C's local-resolution becomes useful once B can install a pinned dorfl), but each is independently buildable; record that ordering as a `blockedBy`/note at tasking time, not a hard gate.
