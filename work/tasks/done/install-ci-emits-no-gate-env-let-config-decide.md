## What to build

`install-ci` emits the advance workflow (`advance-lifecycle`) with a hardcoded `DORFL_*` gate env block — `DORFL_AUTO_BUILD`, `DORFL_AUTO_SLICE`, `DORFL_OBSERVATION_TRIAGE`, `DORFL_SURFACE_BLOCKERS` — baked in with the "calm" values (`true`/`true`/`off`/`false`). The source is `packages/dorfl/src/advance-lifecycle-template.ts` (~L211–214), and its structural validator (~L457–472) currently REQUIRES those env keys to be present.

This is wrong because of how the engine RESOLVES gate policy: `flag > env > per-repo .dorfl.json > global > default` (verified: `resolveRepoConfigFromLoaded` in `repo-config.ts` layers global → per-repo → env → flags). By emitting the env block, install-ci FORCES the env layer to win over the repo's OWN `.dorfl.json`. A user who sets e.g. `surfaceBlockers: true` (or `observationTriage: ask`) in their committed `.dorfl.json` has it SILENTLY OVERRIDDEN by the workflow's `DORFL_SURFACE_BLOCKERS: false` — the env always wins. So per-repo config is dead-on-arrival for any gate the workflow hardcodes, which defeats the whole precedence design and is surprising (the user edits config and nothing changes).

THE PRINCIPLE (decided): the workflow env is the OPTIONAL CI-only OVERRIDE layer, not the carrier of defaults. The default source of truth is the repo's `.dorfl.json` (then global, then the built-in defaults). So `install-ci` should NOT emit gate env that shadows per-repo config; it should let the engine fall through to config/default. A user who genuinely wants a CI-SPECIFIC override (different in CI than locally) adds the env var themselves to their GitHub Actions env — that is the explicit, opt-in CI override the env layer is FOR.

> **DECIDED (maintainer, 2026-06-16) — the design fork is resolved (see `## Decisions`).** CI is treated like ANY OTHER config consumer: the workflow emits NO active gate env at all, so `flag > env > per-repo > global > default` resolves from the repo's `.dorfl.json` (then global, then the built-in defaults). The previously-tricky point is accepted head-on: the built-in defaults are `autoBuild: false`, `autoSlice: false`, `observationTriage: 'off'`, `surfaceBlockers: false` (`packages/dorfl/src/config.ts` ~L451–466), so a CONFIG-LESS repo gets auto-build/auto-slice OFF in CI after this change. **That is the intended, correct behaviour** (the engine is strict / claim-nothing-by-default; CI is not special). A user who wants CI autonomy turns it on the SAME way they would anywhere — by setting `autoBuild: true` / `autoSlice: true` in their committed `.dorfl.json` (governs everywhere) OR by adding the `DORFL_AUTO_BUILD: 'true'` / `DORFL_AUTO_SLICE: 'true'` env var to the workflow themselves (the explicit, opt-in CI-only override the env layer is FOR). To avoid surprising the user with a now-quiet CI, `install-ci` EXPLAINS this at completion (see the Change list). This is the original "let config decide" principle taken to its honest conclusion: the env layer carries no defaults whatsoever.

Change:
- `packages/dorfl/src/advance-lifecycle-template.ts`: STOP emitting ALL FOUR `DORFL_AUTO_BUILD` / `DORFL_AUTO_SLICE` / `DORFL_OBSERVATION_TRIAGE` / `DORFL_SURFACE_BLOCKERS` env lines — BARE OMISSION (decided: Q2 = bare omission; no commented-out example, no dead config to drift). The engine resolves every gate from config/built-in-default. Leave the surrounding non-gate env (`INTEGRATION_MODE`, `SWEEP_MERGED_BRANCHES`, auth, `GH_TOKEN`) untouched, and keep/trim the explanatory header COMMENT so it now says the gate family is resolved from config (not carried here).
- The template's structural VALIDATOR (~L457–472) currently REQUIRES all four gate keys. Flip it: REQUIRE THEIR ABSENCE-as-active env (a regression that re-introduces any active `DORFL_AUTO_BUILD/AUTO_SLICE/OBSERVATION_TRIAGE/SURFACE_BLOCKERS:` assignment must FAIL the validator). Drop the four `require('env-*')` presence clauses and add the absence assertion. Otherwise the validator rejects the corrected template.
- `install-ci` COMPLETION MESSAGE: after the artifacts are written, emit an explanatory line via the existing `log()` sink (`packages/dorfl/src/install-ci.ts` ~L239, right after the `wrote {path}` loop). It tells the user: the emitted advance workflow sets NO gate env, so auto-build/auto-slice are OFF by default (the engine's strict default); to enable CI autonomy, either set `autoBuild`/`autoSlice` in `.dorfl.json` (applies everywhere) or add `DORFL_AUTO_BUILD: 'true'`/`DORFL_AUTO_SLICE: 'true'` to the workflow env (CI-only). Keep it short, copy-pasteable, and accurate to the resolved precedence.

NOTE the relationship to the sibling slice `install-ci-emits-one-advance-workflow-not-redundant-build-slice-tick` (now in `work/in-progress/`): the overlap is SMALL, not the "both rewrite the same file" it first looked like. That sibling's own acceptance says advance-lifecycle's template is NOT modified — it only DELETES the `build-slice-tick` capability + template and removes a dangling HEADER-COMMENT reference to `build-slice-tick-template.ts` inside `advance-lifecycle-template.ts`. So the only shared file region is that one doc-comment, far from the env block this slice edits. The `blockedBy` is therefore a cheap ORDERING courtesy (land the deletion first so this slice rebases against a settled file + does not re-introduce a now-stale comment), NOT a genuine same-region conflict. If the sibling has already landed when this is built, the `blockedBy` is satisfied and there is nothing to coordinate.

## Decisions (resolved by the maintainer, 2026-06-16)

1. **What happens to `AUTO_BUILD` / `AUTO_SLICE` (built-in default `false`, env forces `true`)?** → **CI uses config like any other consumer: emit NO gate env (all four keys), let `flag > env > per-repo > global > default` decide.** A config-less repo therefore gets auto-build/auto-slice OFF in CI (the engine's strict default) — and that is FINE/intended. A user enables CI autonomy by setting `autoBuild`/`autoSlice` in `.dorfl.json` (everywhere) or by adding the `DORFL_*` env var to the workflow themselves (CI-only override). `install-ci` EXPLAINS this at completion so the now-quiet default is not surprising. (This is option (a)-rejected in favour of the cleaner "env carries no defaults" stance — it kills the shadowing for ALL FOUR keys, not just the two lifecycle ones.)
2. **Bare omission vs commented-out example** → **bare omission.** No dead config to drift; the completion message + docs carry the discoverability the comment would have.

### Why this is SAFE to land in THIS repo (and behaviour-IMPROVING, not just neutral)

The dorfl repo's OWN `.dorfl.json` already sets all four gate keys: `autoBuild: true`, `autoSlice: true`, `observationTriage: "auto"`, `surfaceBlockers: true` (verified 2026-06-16). Today the hardcoded workflow env (`AUTO_BUILD:true/AUTO_SLICE:true/OBSERVATION_TRIAGE:off/SURFACE_BLOCKERS:false`) SHADOWS that config, so CI actually runs with `observationTriage` OFF and `surfaceBlockers` FALSE despite the repo asking for `auto`/`true` — exactly the bug the user reported. So this change is behaviour-IMPROVING here: after it, the repo's `observationTriage: "auto"` + `surfaceBlockers: true` finally take effect in CI, and `autoBuild`/`autoSlice` STAY ON (now sourced from config, not env). Critically, **CI does NOT go dark** when this lands in dorfl, because config keeps `autoBuild: true`. (A DIFFERENT, config-less repo WOULD get auto-build/auto-slice off in CI after this — that is the intended strict default, and the completion message tells that user how to opt in.)
