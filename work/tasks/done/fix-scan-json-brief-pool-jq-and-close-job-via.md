But the emitted advance CI matrix `jq` reads `.repos[].prds[]?` + `.cwd.repo.prds[]?`, so it matches NOTHING → the propose matrix enumerates ZERO briefs on the cron (capability B silently dead — the exact failure `ci-propose-matrix-must-enumerate-sliceable-prds-not-only-slices` guards against, inverted by the field rename).

Rename `.prds[]` → `.briefs[]` (and `repos[].prds[]`/`cwd.repo.prds[]` → `...briefs[]`) in:

- `packages/agent-runner/src/advance-lifecycle-template.ts` — the emitted `jq` (line ~314: `(.repos[].prds[]?, .cwd.repo.prds[]?)`) + the validator regex (`/\.prds\[\]/`, ~line 690) + the comments naming `prds[]`.
- `packages/agent-runner/src/advance-ci-template.ts` — the validator regex (`/\.prds\[\]/`, ~line 142) + the comments.
- `packages/agent-runner/src/mirror-pool-scan.ts` — any `.prds` reference.
- The snapshot/assert tests: `advance-ci-template.test.ts` (asserts `\.repos\[\]\.prds\[\]\?`), `advance-lifecycle-template.test.ts`, and `scan.test.ts` if it asserts the key — update to `.briefs[]` / `briefs`.

Rename the close-job discriminator `'prd'` → `'brief'` end-to-end (type, literals, the `cand.via === 'prd'` branch if any, `closeComment`) so it matches the `brief:` field it reads. Update the close-job test asserting `via: 'prd'`.
