> Self-contained ROBUSTNESS / CONCURRENCY-CORRECTNESS slice. It derives from NO SPEC (`covers: []`), so per WORK-CONTRACT.md it omits `prd:` and is its own source of truth.
>
> Source signal: the `gate-on-rebased-tip-fresh-worktree` STOP (its `## Needs attention` block, recorded 2026-06-12). That slice's load-bearing premise (the fresh-worktree gate lands in the SHARED gate-to-integrate band and "benefits `run`" default-ON without regression) does NOT hold for `run`'s CONCURRENT merge-mode integration, because `run` serialises only the CLAIM per repo and treats integration as "fully concurrent" with a plain non-retried `${branch}:main` push. The maintainer chose option (a) from that STOP: a per-repo INTEGRATE lock (NOT a rebase-retry), because it is simpler and more deterministic. This slice is that precursor; landing it lets `gate-on-rebased-tip-fresh-worktree` follow as default-ON.

## What to build

Make `run`'s merge-mode integration genuinely concurrency-safe so two same-repo merge jobs land one-at-a-time instead of relying on benign timing.

Today (verify against current code before building):

- `run` serialises ONLY the CLAIM per repo: `const claimLock = createKeyedLock();` (`src/run.ts` ~L307), threaded into `runOneItem` as `claimLock` (~L338) and applied around the claim (~L484: `ctx.claimLock ? await ctx.claimLock(repoPath, doClaim) : await doClaim()`). The comment block just above the `claimLock` declaration (~L300-306) and the `runConcurrent` block below it (~L312-320) explicitly state that the claim is "the ONLY shared-working-tree step" and that "the agent run + the rebase-or-abort integration stay fully concurrent".
- `run`'s integration goes through the SHARED `performIntegration` core (`src/run.ts` ~L723, `core = await performIntegration({...})`). CRITICAL STRUCTURE (verified): `performIntegration` (`src/integration-core.ts`, `export async function performIntegration` ~L356) runs the WHOLE back-half band in ORDER: `prepare`+`verify` GATE (`runVerify` ~L484), then Gate-2 REVIEW (`reviewGate` ~L548-573), then done-move + COMMIT (~L760), then fetch + REBASE onto `<arbiter>/main` (step 4, ~L778-823), then INTEGRATE (step 5, ~L900+), which on merge mode calls `Integrator.integrate` then `pushBranch(input, '${branch}:main')` (`src/integrator.ts` `integrate`, merge branch). That push is PLAIN non-retried: NO `--force-with-lease`, NO retry. Two concurrent same-repo merge jobs both rebase onto the same pre-merge base and both attempt `${branch}:main`; the loser's push is non-fast-forward. Today it only works by benign timing (the second job usually rebases onto the first's already-pushed merge and fast-forwards).

What to build: add a per-repo INTEGRATE serialiser, reusing the SAME `createKeyedLock` primitive keyed on the repo (the same key the claim lock uses), so same-repo integrations run one at a time.

CRITICAL PLACEMENT (this is the load-bearing design point; do NOT wrap the whole `performIntegration` call): the slow GATE (`prepare`+`verify`) and Gate-2 REVIEW (which can launch a review agent) run at the FRONT of `performIntegration`, BEFORE the rebase-to-integrate tail. Wrapping the entire `performIntegration` call at the `run.ts` site would serialise that expensive gate+review work per repo too, gutting `run`'s parallelism (which is the whole point of `run`). The lock must wrap ONLY the rebase-to-integrate TAIL (step 4 fetch+rebase through step 5 integrate), so the gate and Gate-2 review of multiple same-repo jobs still run CONCURRENTLY, and only the final land-on-main is serialised. So the serialiser is an INJECTED SEAM threaded INTO the core (for example an optional `integrateLock?: <T>(key, fn) => Promise<T>` plus the repo key on `IntegrationCoreInput`), applied around steps 4-5 inside `performIntegration`. The `run` path passes the per-repo lock; single-job callers (`do`/`--isolated`/`--remote`/`complete`) pass none (an un-contended no-op, behaviour unchanged byte-for-byte).

The loser then rebases onto the winner's NOW-ADVANCED main inside the lock before its own push, and its `${branch}:main` is a clean fast-forward. Cross-repo integration stays fully concurrent (the lock is keyed per repo, exactly like the claim lock). This makes two NON-conflicting concurrent same-repo merges both land deterministically (no timing dependence). Two GENUINELY-CONFLICTING same-repo slices cannot both land by construction: the loser's rebase onto the winner's advanced main conflicts and routes ONE to needs-attention (rebase-conflict). That is MORE correct than today's timing-dependent behaviour, where the two slices' merges race and the outcome depends on push ordering.

The lock itself is keyed and threaded in `run` EXACTLY like the existing claim lock so the two are siblings: a second `createKeyedLock()` in `runTick` (the same scope as `claimLock`), threaded through `OneItemContext` into `runOneItem`, then handed to `performIntegration` as the injected `integrateLock` seam (keyed on `repoPath`). Cross-repo concurrency, the calm/registry tick paths, and single-job paths are unaffected (no fleet concurrency to serialise; a single-job path passes no lock).

Update the existing `run` tests that assert "two concurrent same-repo merges both reach done": the contract changes from "two same-repo jobs both integrate" (timing-dependent) to "two NON-CONFLICTING concurrent same-repo merges both land deterministically, and two GENUINELY-CONFLICTING concurrent same-repo merges route exactly one to needs-attention (rebase-conflict)". Keep the non-conflicting-both-land contract green; add or adjust the conflicting case to assert one-done-one-needs-attention. The affected tests (verify these names against the files before editing): `test/run.test.ts` > "two same-repo jobs both integrate under the merge path (claim + integration safe under concurrency)" and "claims at most maxParallel items then stops"; `test/run-loop.test.ts` > "runs TWO same-repo items via the bare mirror CONCURRENTLY (merge), claim+worktree+integration safe". Note: the existing fixtures have each slice write DIFFERENT content to the SAME file via the editing agent (`agent-output.txt`), so two such concurrent slices GENUINELY conflict, and under the new lock the loser correctly routes to needs-attention. To keep a "both land" assertion you need a NON-conflicting fixture pair (each slice edits a DISJOINT file); add that fixture for the both-land case and keep the shared-file pair for the one-conflicts case.

## Decisions (to record while building)

- **Per-repo integrate lock vs bounded rebase-retry.** Lean: the LOCK (a second `createKeyedLock` keyed per repo, mirroring `claimLock`). It is simpler, fully deterministic, and reuses the existing, proven primitive; a bounded rebase-retry is more code, is harder to reason about under contention, and still cannot make two genuinely-conflicting slices both land. Record the choice and the rejected alternative.
- **Genuine conflict routes one to needs-attention.** Record that two same-repo slices which genuinely conflict CANNOT both land; serialising makes the loser rebase onto the winner's advanced main, conflict, and route to needs-attention (rebase-conflict). This replaces today's timing-dependent "both land" with a deterministic, correct outcome.
- **Lock granularity / placement.** Record that the lock wraps ONLY the rebase-to-integrate TAIL inside `performIntegration` (step 4 fetch+rebase through step 5 integrate), NOT the whole `performIntegration` call. Wrapping the whole call would serialise the front-of-band `prepare`+`verify` gate and the Gate-2 review agent per repo, defeating `run`'s parallelism; wrapping only the bare push would leave the loser rebased onto the stale pre-merge base. The correct region is the fetch+rebase+integrate tail, so the loser re-fetches and rebases onto the winner's advanced main inside the lock.
