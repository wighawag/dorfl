### Part A — extend the stale-lease retry to ALL continue-path push sites

`#88` added `pushContinuedBranchWithStaleLeaseRetry` (`src/continue-branch.ts`) and wired it into ONE site: the `createJob` continue path in `src/workspace.ts` (the `run`/`do --remote`/`--isolated` job-worktree path; ~line 243 — anchor on the `pushContinuedBranchWithStaleLeaseRetry(` call, lines drift). But the SAME single `git push <arbiter> <branch>:<branch> --force-with-lease=<branch>` continue-path push — with the SAME latent stale-lease failure — exists UNGUARDED in two sibling sites (per `stale-lease-retry-only-on-job-worktree-path.md`); anchor on SYMBOLS, not line numbers (an active drive shifts them):

- **`src/isolation.ts`** — the in-place checkout strategy's already-pushed-tip reconcile push (the `continued` branch in the strategy, after `rebaseContinuedBranchOntoMain`, ~line 266 — the bare `--force-with-lease=${branch}` push). Conflict signal here: a local `continueRebaseConflict` flag (the strategy surfaces it on its `JobRecord`).
- **`src/start.ts` `continueFromKeptBranch`** (called by `switchToWorkBranch`) — the requeued-branch reconcile push (~line 553 — the `--force-with-lease=${branch}` push). NOTE two DIFFERENCES from the strategy sites, do NOT copy-paste blindly: (i) the conflict signal here is **`rebaseConflict`** on the returned `SwitchResult` (NOT `continueRebaseConflict`); (ii) the push is a **`gitSoft` BEST-EFFORT** push (deliberately non-throwing — its comment: "an unreachable arbiter leaves the local rebased branch; complete's push handles it later").

Factor BOTH through `pushContinuedBranchWithStaleLeaseRetry` so the retry-on-stale-lease applies uniformly. The incident hit one of these uncovered paths (the `--isolated` continue threads through the isolation/job-worktree machinery), so closing them is what makes that exact run self-recover.

**NOT A DROP-IN WRAPPER — thread the helper's REQUIRED inputs (this is the load-bearing detail; do NOT just wrap the bare push).** `pushContinuedBranchWithStaleLeaseRetry` does NOT take the implicit `--force-with-lease=<branch>` the two sites use today; its contract requires the caller to supply an EXPLICIT `expectedRemoteTip` — the arbiter `work/<slug>` sha the caller observed BEFORE the rebase (the pre-rebase remote tip) — plus `mainRef`, and it MUST be called while HEAD is the continued `<branch>`. Mirror the reference site exactly: in `workspace.ts` the pre-rebase tip is captured via `git rev-parse <branch>` against the mirror BEFORE `rebaseContinuedBranchOntoMain`, then passed as `expectedRemoteTip`. So at EACH of the two sites: (1) capture the pre-rebase observed arbiter `<branch>` sha BEFORE the rebase — from **whatever ref actually holds it IN THAT SITE'S CONTEXT** (SAFETY: do NOT blindly `rev-parse <arbiter>/<branch>` — a bare-hub-mirror/job-worktree has NO `refs/remotes/<arbiter>/*` upstream, per the helper's own docstring, so that ref may be EMPTY → a malformed lease; `workspace.ts` reads it via `git rev-parse <branch>` against the MIRROR, the in-place sites read the local checkout's view — find the ref that genuinely resolves to the arbiter's pre-rebase `<branch>` tip in that site, and verify it is non-empty before passing it); (2) confirm HEAD is on `<branch>` (both sites `switch -C` onto it, so this holds — verify); (3) pass `cwd`/`branch`/`arbiter`/`mainRef`/`expectedRemoteTip`; (4) on the helper's `{kind: 'conflict'}` result, set each site's OWN existing conflict signal — `continueRebaseConflict` in `isolation.ts`, **`rebaseConflict` on the `SwitchResult` in `start.ts`** (the names DIFFER — use each site's actual field) — so a conflict routes to needs-attention exactly as the bare path did. A wrong/post-rebase/omitted `expectedRemoteTip` is a wrong lease (retry never fires or re-leases incorrectly) — that is the defect to avoid.

**`start.ts`'s BEST-EFFORT contract — preserve it, but do NOT swallow a real failure (SAFETY).** `start.ts`'s push is `gitSoft` (non-throwing): an unreachable/offline arbiter is TOLERATED because `complete`'s later push handles it. The helper THROWS on a non-stale-lease failure and after the retry cap. So adopting the helper at `start.ts` must preserve the tolerate-offline behaviour WITHOUT re-introducing the silent-strand bug this slice exists to kill. The catch must DISCRIMINATE, not blanket-swallow:
>
> - an **OFFLINE / unreachable-arbiter** failure (the existing tolerated case) → catch + tolerate, exactly as today (the local rebased branch is left; `complete` pushes later);
> - a **real terminal failure** (stale-lease cap exhausted, protected-ref, any non-connectivity error) → do NOT silently swallow — route to needs-attention (Part B) so the work is not silently stranded.
>
> A blanket `try/catch` that drops ALL throws would re-create the exact "sits silently in-progress" bug. Distinguish the two (mirror how `routeToNeedsAttention`'s own best-effort branch-push classifies an outage vs a give-up). Record the chosen discrimination in the `## Decisions` block; do NOT silently convert `start.ts`'s tolerated-offline push into a throwing one, and do NOT silently swallow a real failure.

SEAM CAVEAT (verify, do not assume): `#89` made the `needs-attention/ → backlog/` (requeue) transition tree-less, but the SURFACE direction (`in-progress/ → needs-attention/`) goes through `routeToNeedsAttention` in `src/needs-attention.ts`, which TODAY still takes a `cwd` ("the working clone / job worktree the `work/<slug>` branch lives in") — i.e. it is NOT yet tree-less for surfacing. > ⚠️ **SAFETY — option (a) touches `main` (the one main-write in this slice); follow #89's recipe EXACTLY, do NOT hand-roll it.** `ledgerWrite.applyTransition` pushes `<localBranch>:main --force-with-lease=main:<expectedBase>` — i.e. **`main` IS the push target** (a leased CAS fast-forward, NOT a plain branch push). This is safe ONLY via the precise recipe `#89`/`requeue` uses (see `needs-attention.ts` ~L564–604): (1) FETCH `<arbiter>/main`; (2) `expectedBase` = that JUST-fetched main sha; (3) commit the `git mv in-progress→needs-attention` + reason onto a **THROWAWAY local ref cut OFF the fetched main** (NEVER onto the `work/<slug>` branch — that would push the code-work commits onto main!); (4) `applyTransition({localBranch: <throwaway>, expectedBase, head})`; (5) drop the throwaway ref. A wrong `expectedBase`, or pushing the WORK branch (not a throwaway-off-main) to `:main`, would force-update `main` to the wrong commit — the single most dangerous mistake possible here. If you cannot reproduce that recipe faithfully, take option (b) instead. INVARIANT: the only acceptable `:main` write is the leased CAS fast-forward of a throwaway-off-main ledger commit; NEVER a bare `--force`, NEVER the work branch onto main.

DISTINGUISH the two failure timings (do not conflate):
- Push fails BEFORE the work is committed/done-moved → existing behaviour (whatever it is) is out of scope; this slice is about the AFTER-commit strand.
- Push fails AFTER the work is committed + done-moved on the branch → Part B: surface to needs-attention (tree-lessly if the seam allows), branch (with the green work) left intact on the arbiter, recoverable.
