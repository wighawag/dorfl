## What to build

The CONTRACT phase of the wide refactor (`TASKING-PROTOCOL.md` §3a): now that every caller has migrated onto `spec` (batches 2–4), REMOVE the `prd` alias surface the expand task added, make the dead token dead, and prove the sweep is complete. This is the acceptance GATE for the SOURCE-part cutover.

**SCOPE (option A — source/data split, ratified 2026-07-10; read before building).** The parent spec (§"Solution", US 5/8) splits this cutover into a SOURCE part (dorfl's code + contract SPEAK `spec` — batches 1–5 + this contract task) and a DATA part (dorfl's own `work/prds/* → work/specs/*` folders, `prd:` frontmatter fields, config values, inert refs — owned by the `dorfl prd-to-spec` COMMAND, built next and RUN on dorfl in the FINAL task). So this task's leak scan is IDENTIFIER-SCOPED forward + reverse, NOT a blanket tree grep: it kills the `prd` ALIAS + proves no stray `prd`/`brief` CODE IDENTIFIER leaked, while categorically EXEMPTING the data-folder literals (`work/prds/...`), the `prd:` FIELD token, and domain-PROSE (the artifact word) — those are the command's territory, and the EXHAUSTIVE tree-wide bi-word gate over them (incl. `work/` data) is the FINAL task's, green on dorfl only AFTER the command converts the data. Removing the aliases here does NOT require renaming those data references (they are values/paths/prose, not the removed type-union members).
