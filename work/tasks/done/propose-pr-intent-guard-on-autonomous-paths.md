> Source signal: `work/observations/review-nits-provider-arbiter-derived-and-nopr-intent-2026-06-14.md` (nit 1). The up-front PR-intent failure guard is wired ONLY into the in-place `performDo` path (do.ts step 3c). The autonomous build paths (`performDoRemote`/`runRemotePipeline` for `do --remote`/`--isolated`, and the `run` daemon) do NOT run the probe, so a propose build against a GitHub arbiter with `gh` genuinely unauthed runs the full build and then silently degrades to manual-PR instructions at integration: the exact "wanted a PR, silently didn't get one" failure the `noPR` axis exists to prevent. `--isolated` is the PRIMARY autonomous path (drive-backlog mandates it), so the honest failure is needed MOST there.

## What to build

Extend the existing up-front PR-intent pre-flight guard to the autonomous build paths, reusing the SAME predicate so the in-place and autonomous paths speak with one voice.

Today (verify against current code before building):

- The guard lives ONLY in in-place `performDo` (`src/do.ts` step 3c, ~L658-680): it resolves the arbiter URL (`arbiterUrl(cwd, arbiter, env)`), builds a `gh` availability probe (defaulting to `new GitHubProvider().available(cwd, env)`, injectable via `options.ghCanOpenPr`), and calls `shouldFailProposePrIntent({mode, arbiterIsGitHub, noPR, ghCanOpenPr})` (`src/do-config.ts` ~L378). On a true result it returns `{exitCode: 1, outcome: 'refused', message: PROPOSE_PR_INTENT_GH_UNAVAILABLE_MESSAGE}` BEFORE any claim/onboard/build.
- `shouldFailProposePrIntent` fires ONLY when ALL hold: `mode === 'propose'`, the arbiter is a GitHub URL, `noPR` is UNSET, and the `ghCanOpenPr` probe returns false. CRITICAL: the probe (mirroring `GitHubProvider.available`, which is SYNCHRONOUS, shelling `gh auth status`) is the signal, NOT a config inspection: an ABSENT `providers.github` identity falls back to AMBIENT `gh` auth (the common local-dev case), which the probe correctly reports as available, so a working ambient setup still PROCEEDS.
- `performDoRemote` (`src/do.ts` ~L1437) resolves the hub mirror at ~L1489 (`mirror.url` is the arbiter URL), claims at ~L1588, and runs `runRemotePipeline` (~L1636) which builds + integrates. There is NO probe on this path. `DoRemoteOptions` carries `noPR` and `integration` but NO `ghCanOpenPr` injection seam yet.
- `run` (`src/run.ts`) runs `runOneItem` per item. The per-repo effective config (`config.integration`/`config.noPR`/`config.defaultArbiter`) is resolved at the TOP of `runOneItem` via `resolveRepoConfig` (~L459), and `gitEnv` just after (~L477-482), BOTH BEFORE the CLAIM (~L510). The arbiter URL is derivable from `repoPath` pre-claim the SAME way `claimAgainstRepo` derives it (~L113-121: `git remote get-url origin` on the bare mirror, falling back to `file://${repoPath}`). There is NO probe before the claim/build. CRITICAL: the claim (~L510) happens BEFORE `prepare()` (~L548), so the probe must run BEFORE the claim (not after prepare) or an item would be left CLAIMED (in-progress on main) with no clean home.

What to build: run the SAME up-front `gh` auth/availability PROBE + visible failure on the autonomous paths, BEFORE the build, REUSING `shouldFailProposePrIntent` (do NOT fork it) and `PROPOSE_PR_INTENT_GH_UNAVAILABLE_MESSAGE`.

- **`performDoRemote`/`runRemotePipeline`:** after the mirror is resolved (`mirror.url` known) and BEFORE the claim/build, run the probe and call `shouldFailProposePrIntent({mode: options.integration ?? 'propose', arbiterIsGitHub: isGitHubArbiterUrl(mirror.url), noPR: options.noPR, ghCanOpenPr})`. On a true result return `{exitCode: 1, outcome: 'refused', message: PROPOSE_PR_INTENT_GH_UNAVAILABLE_MESSAGE}` before any claim/build (no wasted work, exactly like the in-place guard). Add a `ghCanOpenPr` injection seam to `DoRemoteOptions` (mirroring `DoOptions.ghCanOpenPr`) so tests can stub `gh` without a real binary; default to `new GitHubProvider().available(probeDir, env)`. Run the probe in a directory that has the arbiter remote (the claim clone dir / mirror), NOT a non-repo cwd.
- **`run` (`runOneItem`):** after `resolveRepoConfig`/`gitEnv` (~L459-482, the per-repo `config.integration`/`config.noPR` are known) and BEFORE the CLAIM (~L510), resolve the arbiter URL from `repoPath` (the mirror's `git remote get-url origin`, falling back to `file://${repoPath}`) and run the same probe. On a true result return a CLEAN PRE-CLAIM item result that does NOT claim, build, or surface a half-built needs-attention (use the existing `config-error`-style failure-cause status, or an equivalently honest non-slice-fault status: the summary already treats `config-error` as a wiring/config failure, not a slice fault). It must NOT crash the tick or abort sibling items (return the item result; `runConcurrent` continues). An injectable probe seam (mirroring the claim/agent seams) so `run`'s tests can stub `gh`. Probing per item is acceptable (the answer is per-repo/per-host; correctness-first matches the in-place guard).

CRITICAL FENCE: the guard must fire ONLY when a PR was genuinely WANTED (propose mode, `noPR` unset) AND `gh` is genuinely unavailable (the auth PROBE returns false, NOT merely an absent identity, which falls back to working ambient auth). MERGE mode, `noPR: true`, a non-GitHub arbiter, and any calm/non-propose path are UNCHANGED (the predicate already returns false for those; do not add new branches). Reuse the existing predicate so the four conditions stay in ONE place.
