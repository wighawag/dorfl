## What to build

Add an `--isolated` flag to the `advance` command, giving `advance` the SAME ISOLATION ERGONOMIC `do --isolated` has — including the FULL variadic grammar (single named item, multiple named items in sequence, AND `-n`/auto-pick): run the advance tick in an ISOLATED job worktree off THIS repo's arbiter (resolved from cwd), then integrate + reap — instead of taking over the current checkout in-place.

Today `advance` has only the two drivers the `advance-loop` SPEC specified: a one-shot IN-PLACE driver (takes over the cwd checkout, refuses on a dirty tree, like bare `do`) and a LOOP driver (`run --advance <mirror>`, isolated + parallel over a hub mirror, the CI shape). There is no isolated form of the ONE-SHOT driver — the exact ergonomic `do --isolated` has. That cell is what lets you run `advance <slug>` (or `advance -n <x>`) from a checkout you are actively working in (a dirty tree), or from anywhere with a participating arbiter remote, without the in-place driver's checkout takeover / dirty-tree refusal — and WITHOUT spinning up the full `run` daemon / a hub-mirror URL. (Confirmed current state: the `advance` action types its flags as `DoFlags` — which already CARRIES `isolated`/`remote` — but never REGISTERS `--isolated` as an option nor READS `flags.isolated`, so `advance --isolated` today fails commander with "unknown option '--isolated'". This slice REGISTERS + WIRES it; it is not a pre-wired flag.)

This is NOT a deep gap — it is a scope/sequencing artifact. The `advance` SPEC was written before `do --isolated` existed as a concept (the SPEC framed isolation as "loop = isolated, one-shot = in-place"); `do --isolated` arrived from a separate observation thread (`do-remote-no-arg-and-remote-autopick` → `do-isolated-in-place`) in the SAME build window, and never got mirrored onto `advance`.

### HONEST SCOPE: what is reused VERBATIM vs what must be BUILT

The isolation *substrate* is reusable; the *per-item execution* is NOT. Do not under-build on a "thin caller, reuse `do`'s pipeline verbatim" premise — the per-item runner is genuinely new. Precisely:

**Reused VERBATIM (the substrate):**

- `resolveArbiterUrlFromCheckout(cwd, arbiterName, env)` in `do.ts` — the `do --isolated` arbiter-URL resolver (re-resolve the arbiter remote from cwd; `undefined` ⇒ the clear "isolated against what?" error naming `--remote <url>`). The ONE true verbatim reuse.
- The job-worktree / hub-mirror isolation pipeline shared by `do --isolated` / `do --remote` / `run` (the `isolation-strategy-seam` + `ensureMirror` + reap-on-failure path) — the worktree + mirror + reap mechanics, NOT what runs inside the worktree.
- The scan→select→refetch SKELETON of `performDoRemoteAuto` (`do-remote-auto.ts`): `ensureMirror` once, scan + `selectPrioritised` the bare mirror's `main` ONCE up front, then a SEQUENTIAL loop where each per-item run re-`ensureMirror`s + RE-FETCHES the same mirror (a fetch, never a re-clone) so item N's worktree branches off a `main` containing item N-1's merge (FRESHNESS — item N rebases onto the latest main; this is NOT re-selection, the selected set is frozen up front). This loop SHAPE is the proven "each step in its own worktree, rebase onto the previous step's merged main" model.

**Must be BUILT (the per-item runner — the real deliverable):**

- An **isolated advance-tick runner**: the per-item unit that runs `performAdvance` (classify → `advancing` lock → surface/apply/triage/build-orchestrate) INSIDE a job worktree off the cwd-resolved arbiter, then integrates + reaps. This does NOT exist today. `do --isolated`'s per-item runner is `performDoRemote`, which runs the full DO pipeline (build/slice only) — it does NOT run the advance tick. So `performDoRemoteAuto` cannot be reused as-is for the per-item step; only its scan/select/refetch SKELETON can.
- The advance auto-pick driver is currently IN-PLACE: `performAdvanceAuto` (`advance-drivers.ts`) uses `scanRepoPaths` (the in-place scan), explicitly "the in-place counterpart of `do`". There is NO mirror-side advance auto-pick driver. The isolated `-n`/auto-pick form needs the advance analogue of `performDoRemoteAuto` (mirror-pool scan + select + the refetch loop, with the isolated advance-tick runner as the per-item unit) — BUILT here, sharing the skeleton, not the per-item runner.

So this slice = REGISTER + WIRE the `advance` CLI `--isolated` branch (reusing the resolver + isolation substrate verbatim) + BUILD the isolated advance-tick runner and its sequential auto-pick driver (sharing only `do`'s scan/select/refetch SKELETON). It still obeys US #26 ("isolation FALLS OUT of the seam") — no NEW isolation MECHANISM is invented; the new code is the advance-specific per-item runner that the existing seam carries.

### Why this composes with all five advance rungs

The isolated advance-tick runner runs `performAdvance` with its cwd pointed at the job worktree; all five rungs then compose with an isolated cwd:

- **build-slice / slice-spec** rungs orchestrate `performDo`, which already has the isolated pipeline — they just run against the isolated worktree's cwd.
- **surface / apply / triage** rungs do tree-less, identity-keyed, ARBITER-resolved CAS writes (the sidecar + `needsAnswers`, the `advancing` borrow) — they target the arbiter regardless of which checkout the tick runs in, so they compose with an isolated cwd unchanged.

(This composition is WHY only the per-item runner is new: the rungs themselves need no change, just an isolated cwd to run in.)

### Scope boundary (mirror `do --isolated` precisely)

- **Full variadic grammar (mirror `do`).** `advance --isolated <slug>` (bare = slice) / `prd:<slug>` / `obs:<slug>` (single); `advance --isolated <a> <b> …` (multiple named, in sequence); `advance --isolated -n <x>` / `advance --isolated` (auto-pick `x`/1 over the mirror-side eligible-pool scan, SEQUENTIAL). The `-n`/auto-pick path runs through the advance analogue of `performDoRemoteAuto` (scan + `selectPrioritised` the pool ONCE up front → sequential loop; each item refetches the mirror so its worktree rebases onto a `main` containing prior items' merges — FRESHNESS). `-n` stays ALWAYS SEQUENTIAL (parallelism is `run`/the CI matrix, US #25); `-n` is mutually exclusive with naming items (the same contract `do` enforces).
- **Snapshot-once, NOT dependency-aware (parity with `do`).** Like `do --isolated -n`, the pool is selected ONCE up front; the loop does not re-scan. A slice `blockedBy:[other]` that is ineligible at scan time is NOT selected and is NOT drained in the same run, even if its blocker is also selected and lands first. This is DELIBERATE parity with `do` — do NOT make `advance --isolated -n` smarter than `do --isolated -n`. Dependency-aware scheduling is a SEPARATE enhancement for BOTH verbs (`work/observations/do-autopick-no-dependency-aware-scheduling.md`), explicitly out of scope here.
- **Orthogonal to `--remote`.** `--isolated` is a boolean naming the ISOLATION intent against THIS repo's arbiter (resolved from cwd); `--remote <url>` is a foreign arbiter. As with `do`, `--isolated` + `--remote` is redundant (a foreign remote is already isolated) → accept it, `--remote` wins. (If `advance` does not yet support `--remote`, keep this slice to the bare `--isolated` form and note `--remote` as out of scope / a sibling.)
- **`--merge` / `--propose` / `--arbiter` resolve identically** to the in-place advance and to `do --isolated` (the same `integrationFromFlags` / arbiter-resolution chain).
- **Distinct from `run --advance`.** `advance --isolated -n` is a one-shot, SEQUENTIAL, single-invocation drain (like `do --isolated -n`); `run --advance <mirror>` is the long-running PARALLEL daemon/CI loop. Both isolate; they are different cadences, not duplicates.
