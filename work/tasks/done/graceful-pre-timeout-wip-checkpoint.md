## Problem (from observation `timeout-minutes-hard-kill-loses-all-wip-no-branch-saved-2026-07-13`)

The per-leg GitHub `timeout-minutes` cap (config `legTimeoutMinutes`, default 120) protects against a wedged leg stranding the run for 6h — but a GitHub `timeout-minutes` is a HARD SIGKILL of the whole runner, so the `do`/`advance` agent never gets to run its work-preserving routing (commit WIP to `work/<slug>` + push). A legitimately-long task (a wide migration, a big refactor) therefore times out, loses EVERYTHING, requeues, and times out again — a permanent trap. Live example: the PR-2b 84-assertion migration (`bounce-migrate-stuck-assertions-and-flip-exit-codes`) ran > 1h10m and risked the hard kill discarding all its work.

## Goal

A dorfl-INTERNAL deadline that fires BEFORE the GitHub hard kill, stops the agent GRACEFULLY, saves the WIP to `work/<slug>`, and — crucially — **AUTO-REQUEUES (continue) WITHOUT a human sidecar when the leg made progress**, so a legitimately-long task drains across ticks fully autonomously. The GitHub `timeout-minutes` stays ONLY as a last-resort backstop. This turns "over-cap task loses all work forever" into "over-cap task makes incremental, autonomous progress across runs".

### A deadline checkpoint is NOT a failure (the core semantic decision)

A hard-kill-avoided checkpoint means "healthy work in progress, ran out of time" — NOT "something went wrong, a human must look". So it MUST NOT surface a `needsAnswers:true` question sidecar the way `saveAgentFailure` does; that would put a human in the loop for every long task, defeating autonomy. Instead the checkpoint AUTO-CONTINUES: save the WIP to the branch, RELEASE the lock (leave the branch), and let the NEXT tick's claim continue from the branch tip. No human, no sidecar — a self-draining loop.

### The anti-loop guard (the safety catch that makes auto-requeue safe)

Unconditional auto-requeue would let a GENUINELY WEDGED agent (making no progress) checkpoint → auto-requeue → wedge → forever, burning CI compute — WORSE than today's "loses work" trap (which at least stops). So auto-continue is GATED on PROGRESS + a CEILING:

- **Made progress ⇒ auto-continue.** The checkpoint's WIP commit must DIFFER from the branch tip the leg started from this session (the agent actually changed something). Reuse the EXISTING empty-diff progress primitive `isWorkBranchDiffEmpty` (`do.ts` ~line 1559, from task `noop-backstop-counts-branch-commits`): a non-empty branch diff this session ⇒ healthy ⇒ auto-continue.
- **No progress this session ⇒ SURFACE to a human** (the `needsAnswers:true` sidecar, via the normal bounce path). A checkpoint that produced NO new work is indistinguishable from a wedge — a human should look ("this leg made no progress before the deadline; is it stuck or mis-scoped?").
- **Ceiling ⇒ SURFACE to a human.** Track a small auto-continue counter (on the lock entry / branch); after `maxAutoCheckpoints` (default a small N, e.g. 5) consecutive deadline auto-continues, STOP auto-requeuing and surface a `needsAnswers:true` question ("this task has hit the deadline N times — it may be too big for one CI leg; split it or run it locally?"). This bounds even a slowly-progressing-but-never-finishing task and is the signal a task is mis-scoped (exactly PR-2b's situation). The counter RESETS on any non-deadline outcome (a normal completion, a real bounce).

So the routing decision at the deadline is: **progress AND under-ceiling ⇒ auto-continue (release lock, no sidecar); else ⇒ surface a needs-answers question (human decides continue / split / cancel).**

Two-layer deadline model (the maintainer's framing):

- **dorfl-internal deadline (the PRIMARY stop): default 1h (60 min).** The agent session self-stops at this budget; dorfl then SAVES the WIP (commit + push `work/<slug>`) and routes it per the progress+ceiling decision (auto-continue, or surface) with the remaining head-room. This is the deadline that should normally fire. (1h fails a genuinely-wedged leg FAST while a healthy long task just auto-continues across more ticks — cheap now that auto-requeue is the design.)
- **GitHub `timeout-minutes` (the BACKSTOP): must be LARGER than the dorfl deadline** to leave room for the leg's PRE-steps (checkout, dorfl-setup, install) and POST-steps (the save/push itself, reap). Render it as `dorfl-deadline + headroom` (dorfl 60 + a `checkpointHeadroomMinutes` margin, default 30 ⇒ GitHub 90 = 1h30), so the GitHub kill only fires if the graceful save ITSELF wedges. NEVER let GitHub's cap be ≤ the dorfl deadline (then the backstop pre-empts the graceful path — the bug we are fixing).

## Design (grounded in the code — verify at build, STOP if a premise is false)

The seam is `runDoAgent` (`do.ts`) → `launchWithOptionalWatch` (`agent-launch.ts`) → `PiHarness.launchAsync` (`pi-harness.ts`), which spawns pi via `spawn(...)` and resolves its Promise on the child `exit`. A deadline attaches HERE:

1. **Thread a deadline into the launch.** Add an optional deadline/abort input to `LaunchInput` (e.g. `deadlineMs?: number` or an `AbortSignal` — pick one; a signal composes better and is more testable). Resolve the effective deadline in `do`/`advance` from config (see the config knob below) and pass it through `runDoAgent` → `launchWithOptionalWatch` → the harness.
2. **Race the launch against the deadline in `launchAsync`.** When the deadline fires: send `SIGTERM` to the pi child (a graceful stop; pi flushes its session `.jsonl`), then `SIGKILL` after a short grace (e.g. 10s) if it has not exited. The Promise resolves with a `LaunchResult` that flags the deadline stop (a new field, e.g. `timedOut: true`, distinct from `ok`/`detail`) so the caller can route it as a CHECKPOINT, not a generic failure. Preserve the existing exit/close/error settle-once discipline (do not double-settle the Promise); reuse the same stdout-drain + FD-release the current `exit` handler does.
3. **Route the checkpoint by the progress+ceiling decision — SPLIT save from stuck.** In `do.ts` (and `run.ts`'s mirror), when the launch reports the deadline stop:
   - **ALWAYS SAVE the WIP first**: commit the tree + push `work/<slug>` to the arbiter. Reuse the SAVE HALF of the existing mechanism (the wip-commit + branch-push in `applyNeedsAttentionTransition` / the `--reset`-less branch preservation) — do NOT invent a new save.
   - **AUTO-CONTINUE branch (progress AND under ceiling): release the lock, KEEP the branch, NO sidecar.** This is `returnToBacklog`'s DEFAULT keep+continue path (`needs-attention.ts`: "lock released, branch preserved"; the next claim continues from the tip, rebased onto fresh main at onboard). Do NOT call `saveAgentFailure` here — it MARKS THE ITEM STUCK on its lock (`do.ts`: "marked '<slug>' stuck on its lock"), which is the OPPOSITE ledger outcome of auto-continue. `saveAgentFailure` is for the FAILURE path only. Increment the auto-checkpoint counter; exit 0.
   - **SURFACE branch (no progress this session, OR ceiling hit): the stuck/needsAnswers path.** THIS is where the `applyNeedsAttentionTransition` surface (set `needsAnswers:true` + write the question sidecar, human decides continue/split/cancel) belongs — the only path that pulls a human in. Its reason names "deadline checkpoint (no progress / ceiling)".
   - The reason is DISTINCT per branch ("deadline checkpoint (auto-continued N/max)" vs "deadline checkpoint (no progress / ceiling)"), never a generic agent-failed.
   - RESUME: because auto-continue KEEPS the branch, the next claim (a plain `dorfl advance`/`do`) continues from its tip — the DEFAULT keep+continue behaviour, NO special flag. (`requeue` is only needed to recover a STUCK item; an auto-continued checkpoint is not stuck.)
4. **Config is the SINGLE source of truth (`dorfl.json`), read at RUN TIME by BOTH consumers — so editing it reflects everywhere with NO `install-ci` re-run.** Add resolved config fields, each through the SAME flag > env > per-repo > global > default chain the other knobs use, with FAIL-LOUD coercion (throw on out-of-range, like the existing env-config coercion — NEVER silently clamp):
   - `agentDeadlineMinutes` — the internal deadline. Default 60; MAX 240 (4h) (fail-loud over max).
   - `checkpointHeadroomMinutes` — the GitHub-backstop margin above the deadline. Default 30; MIN 10, MAX 60 (1h) (fail-loud out of range). The MIN 10 guarantees the backstop is always ≥ 10 min above the deadline, so a graceful save always has room.
   - `maxAutoCheckpoints` — the auto-continue ceiling. Default e.g. 5.

   **Both consumers read the SAME committed `dorfl.json` at run time:**
   - the `advance`/`do` LEGS read `agentDeadlineMinutes` via normal config resolution → the internal deadline (step 1);
   - the `enumerate` JOB emits the GitHub cap dynamically (below).

5. **A NEW `dorfl config --json` command + a DYNAMIC GitHub `timeout-minutes`.** Do NOT bake a static number into the workflow YAML (today's `timeout-minutes: ${config.legTimeoutMinutes}`), because that only changes on an `install-ci` re-run — the drift we are removing. Instead:
   - Add `dorfl config --json`: prints the RESOLVED config (the normal precedence chain) as JSON. A focused honest primitive — do NOT overload `scan`.
   - In the advance-lifecycle template's `enumerate` job (which already runs `dorfl scan --json | jq` and emits job outputs), add a step that reads `dorfl config --json` RUN IN THE CHECKOUT (so it resolves THIS repo's COMMITTED `dorfl.json`, exactly like `scan --here` — not a global/default layer), computes `githubTimeout = agentDeadlineMinutes + checkpointHeadroomMinutes`, and emits it as a job OUTPUT (via `$GITHUB_OUTPUT`, exactly like `items`/`any`).
   - Change ONLY the AGENT-LEG jobs (`advance-propose`, `advance-merge`) to `timeout-minutes: ${{ needs.enumerate.outputs.githubTimeout }}` (dynamic, read at run time from the committed `dorfl.json`). Leave the `enumerate` + `reap-merged-branches` job timeouts UNTOUCHED (they have none today; not this task's concern).
   - Net: a user edits `agentDeadlineMinutes`/`checkpointHeadroomMinutes` in `dorfl.json`, commits, and the NEXT tick uses it for BOTH the internal deadline AND the GitHub backstop — no `install-ci` re-run, no baked-in YAML number, no drift.

   REMOVE `legTimeoutMinutes` + its `--leg-timeout-minutes` install-ci flag entirely (the resolved decision above): the dynamic render replaces the only thing they drove. `install-ci` MAY keep writing the NEW fields' defaults into `dorfl.json`, but it is NO LONGER the authority for the running value — the committed config is. Update the install-ci template's `require(...)` self-checks and any tests that asserted the old static `timeout-minutes: ${config.legTimeoutMinutes}` render to the dynamic form.

## Non-goals / boundaries

- Do NOT try to hook GitHub's SIGKILL (there is no graceful-shutdown hook; that is exactly why the internal deadline exists). The GitHub cap stays a dumb backstop.
- Do NOT add a resumable "continue-token" protocol (the observation's option 3) — the branch push IS the durable checkpoint; the next claim's DEFAULT keep+continue already resumes from the branch tip. Out of scope.
- Do NOT change what a normal (non-deadline) agent run does — a run that finishes before the deadline is byte-for-byte unchanged (the deadline timer is cleared on a normal `exit`).
- The checkpoint saves the tree AS-IS (possibly mid-edit / not green). That is fine and intended — it is recoverable WIP, exactly like the agent-failure save; do not gate the checkpoint on a green tree.

## Acceptance criteria

- [ ] A launch that exceeds the internal deadline is stopped with SIGTERM (then SIGKILL after a grace) and the child's partial WIP is COMMITTED to `work/<slug>` and PUSHED to the arbiter (test end-to-end with an injected fast deadline + a stub/slow agent).
- [ ] AUTO-CONTINUE on progress: a deadline checkpoint whose WIP DIFFERS from the session-start branch tip, under the ceiling, RELEASES the lock (item eligible again), leaves the branch, writes NO `needsAnswers` sidecar, and returns `exitCode: 0`; the next claim continues from the branch tip (test: injected deadline + agent that made a change ⇒ no sidecar, lock released, branch carries the WIP).
- [ ] SURFACE on no-progress: a deadline checkpoint that made NO new work this session surfaces a `needsAnswers:true` question (human decides), NOT an auto-continue — the anti-loop guard (test: injected deadline + agent that changed nothing ⇒ sidecar surfaced).
- [ ] CEILING: after `maxAutoCheckpoints` consecutive deadline auto-continues the checkpoint STOPS auto-requeuing and surfaces a `needsAnswers:true` question ("hit the deadline N times — split or run locally?"); the counter resets on any non-deadline outcome (test: drive the counter to the ceiling ⇒ surface).
- [ ] The checkpoint reason is DISTINCT ("deadline checkpoint (auto-continued N/max)" vs "deadline checkpoint (no progress / ceiling)"), never a generic agent-failed — verified in the recorded reason.
- [ ] A launch that finishes BEFORE the deadline is byte-for-byte unchanged (the timer is cleared; no SIGTERM, same `LaunchResult`) — a control test.
- [ ] `dorfl config --json` prints the RESOLVED config as JSON (through the normal precedence chain); a test covers it (does NOT overload `scan`).
- [ ] DYNAMIC GitHub timeout: the advance-lifecycle template's `enumerate` job emits `githubTimeout = agentDeadlineMinutes + checkpointHeadroomMinutes` (read from `dorfl config --json` at run time) as a job output, and ONLY the agent-leg jobs (`advance-propose`/`advance-merge`) set `timeout-minutes: ${{ needs.enumerate.outputs.githubTimeout }}`; a template test asserts the dynamic reference (NOT a baked-in number) and that `enumerate`/`reap` timeouts are untouched.
- [ ] SINGLE SOURCE: editing `agentDeadlineMinutes`/`checkpointHeadroomMinutes` in `dorfl.json` changes BOTH the runtime internal deadline (the legs) AND the rendered GitHub cap (the enumerate output) with NO `install-ci` re-run — verified via the config-resolution + the enumerate-output computation.
- [ ] The three fields resolve through the standard flag > env > per-repo > global > default chain (defaults: deadline 60, headroom 30, maxAutoCheckpoints 5); a config test covers the precedence.
- [ ] FAIL-LOUD coercion (throws, never silently clamps): `agentDeadlineMinutes` > 240 (4h), `checkpointHeadroomMinutes` < 10 or > 60 (1h) each raise a clear config error; the MIN-10 headroom guarantees the GitHub backstop is always ≥ 10 min above the deadline. A config test covers each bound.
- [ ] `legTimeoutMinutes` + its `--leg-timeout-minutes` install-ci flag are REMOVED; no config field, CLI flag, template render, or test references them; the ADR recording the vocabulary change (deadline/backstop replaces `legTimeoutMinutes`) is written.
- [ ] Both `do` (in-place) and `run` paths honour the deadline + checkpoint (the shared save path already spans both — verify the mirror).
- [ ] Acceptance gate green: `pnpm -r build && pnpm -r test && pnpm format:check`.
