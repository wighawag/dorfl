## What to build

Gate the autonomous stranded-done auto-recover so it can no longer SILENTLY DISCARD a continue-agent's new work. Today the auto-detect in `complete.ts` sets `committedRecovery` purely from BRANCH FOLDER STATE (`!onInProgress && !onNeedsAttention && onDone`). The recover path (`recoverAlreadyCommitted` in the integration core) SKIPS the `git add -A` + commit step and only rebases+integrates the ALREADY-committed kept tip. So on a CONTINUE (a requeued slice whose prior attempt already done-moved the slice into `done/` on the kept branch), the predicate fires even when the agent just produced NEW, uncommitted work this run — and that work is silently discarded.

Gate: `committedRecovery` may be true ONLY when this run produced NO UNCOMMITTED WORKING-TREE work AND the branch is done-stranded. The precise disambiguator is **whether the WORKING TREE is dirty with new source edits this run** — i.e. `git status --porcelain` over non-`work/` source paths (catching the agent's unstaged + untracked edits, excluding the runner's job-record), the SAME working-tree predicate the `isWorkBranchDiffEmpty` helper's FIRST half uses (`src/agent-stop.ts` runs exactly that porcelain check with the `:(exclude)work` + job-record exclusions). Dirty working tree ⇒ the agent produced new work this run ⇒ do NOT recover.

- Downstream, `integration-core.ts` step-2 `git mv work/in-progress/<slug>.md → work/done/<slug>.md` cannot run either (the slug is already in `done/`). The whole build-path source contract (`IntegrationCoreInput.source`, the step-2 `git mv`, the `existsSync(sourcePath)` originTrust read, `reconcileDoneMoveAgainstArbiter`) bakes in "the slice is moved INTO done/ for the FIRST time on this commit." A continue-from-done-strand is structurally outside it.

AUTO-LANDING a dirty continue therefore needs a NEW lifecycle path (a `source: 'done'` / `alreadyDoneMoved` contract that skips the `git mv` and exempts originTrust + reconcile) — a real, reviewable design change that is OUT OF SCOPE here. It is owned by the humanOnly sibling slice **`complete-builds-on-already-done-moved-continue`**, which is `blockedBy` THIS slice.

THIS slice is the URGENT data-loss STOPPER only: a dirty continue must be **NEVER silently discarded**. Where the gate now prevents the silent recover, the resulting refusal must surface to `needs-attention/` with a CONTINUE-SPECIFIC, actionable reason (NOT the bare `nothing to complete`) so the human knows what happened and how to finish it. The clean genuine-strand recover (story 2), the explicit `complete --isolated` recover (story 3), and the already-integrated no-op all stay UNCHANGED.

> CRITICAL — TWO traps, both verified against the code:
>
> 1. Do NOT use the core's `nothingStaged` helper. It is `git diff --cached --quiet` (INDEX only) and works in the core ONLY because the core calls it AFTER `git add -A` (integration-core.ts step 3). At the `complete.ts` auto-detect point (BEFORE `performIntegration`) the agent's edits are UNSTAGED, so `nothingStaged` reads "nothing staged" = true and the recover would STILL mis-fire — the bug survives. Use the WORKING-TREE porcelain check, not the index check.
>
> 2. Do NOT gate on the FULL `isWorkBranchDiffEmpty(...)`. Its SECOND half (`hasSourceCommitsAhead`: counts non-`work/` commits in `<arbiter>/main..HEAD`) returns "has work" for a GENUINE STRAND too — a stranded-done branch by definition carries its implementation SOURCE COMMITS ahead of main. So `!isWorkBranchDiffEmpty` is true for BOTH a dirty continue AND a clean genuine strand, which would WRONGLY BLOCK the legitimate recover (breaking story 2). Use ONLY the WORKING-TREE-dirty half (the uncommitted-edits porcelain check), NOT the commits-ahead half. If a shared helper is desired, EXTRACT the working-tree-dirty predicate (the porcelain half) so both `isWorkBranchDiffEmpty` and this gate share it, rather than calling the composite.

The result:

- **Dirty tree (agent produced work)** ⇒ recover is NOT taken; instead of the silent discard, the run surfaces to `needs-attention/` with a continue-specific reason (the kept branch already holds `<slug>` in `work/done/` and this run produced uncommitted edits — finish with `complete --isolated <slug>` after committing, or `requeue --reset` and rebuild). The new work is NEVER silently thrown away. Auto-LANDING it is the sibling slice.
- **Clean tree + done-stranded tip (a genuine finished strand)** ⇒ recover the kept commit (the original `autonomous-path-auto-recovers-already-committed-stranded-branch` behaviour, preserved).

This keys off the WORKING-TREE-dirty state (the porcelain half only, per the traps above), needs NO agent signal and NO onboard-decision threading. The explicit `complete --isolated <slug>` surface (which deliberately recovers a stranded worktree and sets `committedRecovery` directly via `recover-isolated.ts`) is UNCHANGED — this narrows only the autonomous auto-detect path.

## Needs attention

The slice rests on a premise that the current code contradicts. The slice says: gate `committedRecovery` on a clean working tree, and the DIRTY continue case "MUST take the normal build→commit→integrate path so the work lands". But on a CONTINUE the slice file is ALREADY in `work/done/` on the kept branch (that is the whole reason the folder-shape auto-detect mis-fires today — the slice itself describes this state). And the "normal build path" in `src/complete.ts` (~L478–484) REFUSES exactly that state:

  const source: 'in-progress' | 'needs-attention' = onInProgress ? 'in-progress' : 'needs-attention';
  const sourcePath = source === 'in-progress' ? inProgress : needsAttention;
  if (!committedRecovery && !existsSync(sourcePath)) {
    throw new CompleteRefusal(`work/in-progress/${slug}.md (nor work/needs-attention/${slug}.md) found — nothing to complete (already done, or wrong slug?).`);
  }

So with the slice's described fix applied in isolation (`committedRecovery = folderShapeStranded && !hasUncommittedSourceChanges(...)`), a dirty done-stranded continue stops being SILENTLY DISCARDED (good) but is then NOISILY REFUSED with "nothing to complete" (does NOT satisfy acceptance criterion 1: "the new work is committed + done-moved + integrated, NOT discarded"). I verified this end-to-end: the slice-1 / isolated / already-integrated tests stayed green; the new dirty-continue regression test (writeFileSync of an unstaged edit + a new untracked file before performComplete on `seedStrandedDoneBranch('delta')`) returns `outcome: 'refused'` with that exact message.

Downstream, `integration-core.ts`'s step-2 done-move would also fail on the dirty path (`git mv work/in-progress/<slug>.md → work/done/<slug>.md` cannot run when in-progress/ no longer holds the slug — it is already in done/). The build-path source-resolution contract (`source: 'in-progress' | 'needs-attention'` in `IntegrationCoreInput`, the explicit `git mv` in step 2 ~L778, the `existsSync(sourcePath)` originTrust check ~L534, and `reconcileDoneMoveAgainstArbiter`) all bake in "the slice is being moved from in-progress/ or needs-attention/ INTO done/ for the first time on this commit". A continue-from-done-strand is structurally outside that contract.

The slice is therefore INCOMPLETE — it specifies the gate but assumes a normal-build-path that does not actually exist for the dirty done-stranded case. Resolving this is a load-bearing design choice that touches contracts/concepts used across `complete`, `do --remote`, `run`, and the slicing lifecycle seam, well outside the "extract a porcelain predicate + flip one boolean" scope this slice describes. Possible re-scopes (each a real DESIGN decision, not a small factual gap, hence STOP rather than silent choice):

  (A) Extend `complete.ts` source resolution to accept `work/done/<slug>.md` as a valid source when `folderShapeStranded && hasUncommittedSourceChanges(...)`, AND extend `IntegrationCoreInput` with `source: 'done'` (or an `alreadyDoneMoved: true` flag) that makes the integration core SKIP the `git mv` step (the file is already in done/), still run prepare→gate→`git add -A`→commit→rebase→integrate, and exempt this state from the originTrust read + the `reconcileDoneMoveAgainstArbiter` "where did the slug move?" reasoning. This creates a third state alongside the existing `recovering` (needs-attention re-finish) and `committedRecovery` (clean strand fast-path): a "continue-build" lifecycle. New concept, NEW user-visible surface — must be named coherently with the existing CONTEXT.md vocabulary and reviewed.

  (B) Use the existing `IntegrationLifecycle` seam to inject a no-op `stage()` plus `titlePath = work/done/<slug>.md` from `complete.ts` for the dirty done-stranded case. This works on the move side but silently changes load-bearing behaviour: lifecycle is treated as "non-slice" throughout the core (`!lifecycle` gates the untrusted-origin originTrust check ~L531, the sibling-slug arbiter reconciliation ~L873/L911, the `recovering` recovery rebase ~L895, and the `commitTag: 'done'` default). Calling a slice build a "lifecycle" muddles the concept and re-means the seam.

  (C) In `complete.ts`, when `folderShapeStranded && dirty`, pre-`git mv work/done/<slug>.md → work/in-progress/<slug>.md` BEFORE `performIntegration` so the existing build path can do its normal done-move. The two moves cancel in the index; the `git add -A` captures the agent's edits. Contained but introduces a hidden mutate-then-restore the existing integration-core does not know about, and the reconciliation step's "where did the slug move?" arbiter check now sees a no-net-move it would not otherwise expect.

  (D) Re-scope the slice as the NARROWER "stop the data loss; refuse loudly instead": accept that the dirty done-stranded continue surfaces as a noisy `CompleteRefusal('nothing to complete')` and route to needs-attention with a continue-specific message ("the kept branch already has '<slug>' in work/done/ and your run produced uncommitted edits — finish this with `complete --isolated <slug>` after committing, or `requeue --reset` and rebuild"). Then a SIBLING slice owns "land a dirty continue automatically" with the contract change of (A).

My read is (D) — strictly stop the data loss in THIS slice, hand off "land the dirty continue automatically" to a sibling — is the smallest coherent re-scope, but it changes acceptance criterion 1 (which currently demands the new work be integrated, not just safely refused). That is a slice-author / reviewer call, not mine to make silently mid-build.

Suggested re-scope for this slice: NARROW criterion 1 to "the dirty continue is NEVER silently discarded — it either lands OR routes to needs-attention with a continue-specific reason" + add a sibling slice for the build-path contract change of (A). With that re-scope my surgical implementation (extract `hasUncommittedSourceChanges` in `agent-stop.ts`, gate `committedRecovery = folderShapeStranded && !hasUncommittedSourceChanges(...)` in `complete.ts`, plus a regression test pinning "dirty continue does NOT silently auto-recover; clean strand still does") is the complete fix and is ready to land.
