## What to build

The `work/needs-attention/` FOLDER and its `git mv`-into-it move are RETIRED:
a stuck item is now recorded as `state: stuck` on its per-item lock ref, not
moved into a folder. But the code and user-facing prose still describe the old
folder move as live. This task removes the dead folder machinery end-to-end and
makes every human-facing string and comment describe the current lock-state
model, so a user is never pointed at a `work/needs-attention/<slug>.md` path
that does not exist.

End-to-end, the surviving behaviour after this task:

- A stuck item surfaces by having `state: stuck` (with its reason) recorded on
  its per-item lock ref. There is NO `work/needs-attention/` directory and NO
  move into one.
- `requeue` recovers a stuck item by clearing/transitioning that lock state and
  moving the slug from its ACTUAL current folder (`in-progress/`) back to the
  backlog, as a tree-less CAS to the arbiter ref. It no longer recovers "from
  `work/needs-attention/<slug>.md`".
- All runtime notes the user sees, and the `requeue` help text, name the
  lock-state model, not the folder.

Scope is a CLEAN-BREAK removal of the retired path, not a compatibility shim:
delete the dead move/surface helpers and the folder branch of the recover path
rather than leaving them dual-wired.
