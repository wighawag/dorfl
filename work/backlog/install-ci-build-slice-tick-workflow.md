## What to build

The `install-ci` capability that emits the **build/slice tick** workflow — capabilities A (auto-build ready slices) and B (auto-slice ready PRDs) — on top of the shared core. CI always invokes **`advance`** (a strict superset of `do`; with the lifecycle gates at their calm defaults it degrades to `do`'s build/slice behaviour — ADR `ci-config-policy-and-gate-family`), tuned by the `AGENT_RUNNER_*` env block. This slice wires the cron + `workflow_dispatch` triggers and the config-driven integration mode, in-place in the CI checkout (no isolation machinery).

End-to-end path:

- Emit ONE fixed workflow file for the build/slice tick, generated by the GitHub adapter from the shared core, triggered by **cron** (scheduled drain) + **`workflow_dispatch`** (manual catch-up/debug with an `integrationMode` input).
- Encode the **propose=matrix / merge=sequential** discipline verbatim from the existing seed (validated by `src/advance-ci-template.ts`): `propose` ⇒ a DYNAMIC matrix (`agent-runner scan --json | jq` emits the deduplicated eligible list, one leg per id, each leg carries `--propose` so it can NEVER merge to `main`); `merge` ⇒ a SINGLE SEQUENTIAL `advance -n <x> --merge`. ONE word `integrationMode` drives BOTH the job shape AND the integration flag so they cannot desync.
- Use the engine gate family via the workflow's `AGENT_RUNNER_*` env block (`AGENT_RUNNER_AUTO_BUILD`, `AGENT_RUNNER_AUTO_SLICE`), with both lifecycle question-gates at their calm defaults (this is "calm build-only `advance`", NOT a different verb). No new `autoAdvance` gate.
- CI runs IN-PLACE in the checkout (the CI container IS the isolation; no `--isolated`/`--remote`/registry). Add a CI **concurrency group** so overlapping ticks never collide; the claim CAS is the real cross-run serialiser.
- CI / automation uses explicit slug prefixes (`advance slice:foo` / `advance prd:foo`), NEVER bare (ADR command-surface-and-journeys §3a). (CI always invokes `advance`, never `do` directly — see below.)
- The generated artifact is tested by emitting into the `--fake` scratch dir and asserting on the produced YAML (reuse `src/advance-ci-template.ts`'s structural validator); no live Actions run.
- **File-orthogonality:** add this capability as a NEW self-registering emitter module via the core's capability-registry seam (from `install-ci-core-and-github-adapter`) — do NOT hand-edit a shared central list/switch, so this slice and the other three capability workflow slices (build-tick, advance-lifecycle, intake, close-job 2014 minus this one) stay mergeable in parallel.

**Gate (agent-buildable):** this slice BUILDS a deterministic generator, snapshot-tested under `--fake`; it does NOT itself land a live workflow, set a real secret, or change a repo permission (the human runs `install-ci` and commits the output, and the running CI job is forbidden from editing `.github/workflows/**` — US #9). The security-sensitivity lives in the generated ARTIFACT at runtime, gated by the human-run scaffolder + the runtime config gates, not by this slice's build. So no `humanOnly` (the PRD-level `humanOnly` does not propagate; the slice gate is decided from building THIS slice).

## Acceptance criteria

- [ ] `install-ci` emits a single fixed build/slice-tick workflow that invokes `advance` (NEVER a user-chosen verb), triggered by cron + `workflow_dispatch` (the dispatch carries an `integrationMode` input).
- [ ] `integrationMode=propose` produces a DYNAMIC matrix (enumerated via `scan --json`, one leg per eligible id, each leg passes `--propose`); `integrationMode=merge` produces a SINGLE SEQUENTIAL `advance -n <x> --merge`. The one `integrationMode` value drives BOTH shape and flag (no way to desync).
- [ ] The generated workflow exposes the gate family via an `AGENT_RUNNER_*` env block (at least `AGENT_RUNNER_AUTO_BUILD` / `AGENT_RUNNER_AUTO_SLICE`); the two question-gates sit at their calm defaults so out-of-the-box behaviour is build/slice-only with no questions.
- [ ] The job runs IN-PLACE (no `--isolated`/`--remote`/registry) and carries a CI concurrency group; the fully-autonomous-to-main path (all gates on + `--merge` everywhere) is a loud, non-default opt-in, never the default.
- [ ] All invocations use explicit slug prefixes (`slice:` / `prd:`), never bare.
- [ ] The running CI job NEVER edits `.github/workflows/**` (US #9): the emitted workflow needs no `workflows` permission and cannot rewrite its own triggers.
- [ ] Tests generate into the `--fake` scratch dir and snapshot/structurally-validate the produced YAML (reuse `src/advance-ci-template.ts`); no live Actions run, no network, stubbed `GitHubCIContext`.
- [ ] **Shared-write isolation:** `--fake` writes the workflow to `.fake/`, never a real `.github/`; tests assert the real `.github/` and any real secrets store are untouched after the run.
