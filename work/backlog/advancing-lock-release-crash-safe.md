## What to build

Make the `advancing` lock release land after a FAILING post-lock dispatch (recover / integrate / gate), so a failed `advance` run never leaves an orphaned advancing marker and a one-slug-two-folder ledger.

> ROOT-CAUSE FIRST — the exact failure mechanism is NOT yet proven; REPRODUCE it before fixing (do not trust the hypotheses below). Verified facts that NARROW it: (a) `recoverAlreadyCommitted` does NOT throw on a conflict — it runs `git rebase --abort` and RETURNS `outcome: 'rebase-conflict'` (integration-core.ts ~L1416); (b) `performComplete` "never throws for the rebase-conflict case" — it returns exit 1; (c) so the `exec` result returns normally and `advance`'s `finally { release(item) }` (advance.ts ~L903) DOES run. Therefore the release was NOT skipped by a throw — it RAN and FAILED INTERNALLY to land (the incident left the marker orphaned with no `advancing: release` commit). The FIRST task is a failing test that reproduces the orphaned marker through the REAL `advance` + real `releaseAdvancingLock` (not the stubbed `acquireLock`/`releaseLock` that `advance.test.ts` uses today) and IDENTIFIES why the release did not land. Leading hypotheses to confirm/refute: the release's top dirty-guard (`git diff --quiet` / `--cached`) trips because after the recover's rebase-abort the worktree differs from HEAD (the done-moved slice file in `work/done/`, and/or the `work/advancing/<entry>.md` marker the acquire wrote into the tree); OR `cleanup`'s `git checkout <origRef>` refuses on that state; OR a CAS contention/`lost` against the just-advanced main. Fix the ACTUAL cause found; do not ship the abort-based fix below if the repro shows a different cause.

`advance` takes the `advancing` CAS borrow (`work/advancing/<entry>.md`, `<entry>` = `<type>-<slug>`) and releases it in a `finally`. The release mechanism is ALREADY self-isolating and work-preserving: `runRelease` captures the current ref (`originalRef`), `releaseAttempt` does its marker-removal CAS on a SCRATCH `advancing-release/<entry>` branch cut from `<arbiter>/main` (never the work branch), and a `finally` `cleanup` does `git checkout <origRef>` + deletes the scratch branch — so the work branch tip is checked back out, never reset. The ACTUAL blocker is the guard at the TOP of `runRelease` that THROWS when the working tree/index is dirty ("commit/stash them before releasing") BEFORE any of that runs; and even past it, `git checkout` of the scratch/orig refs REFUSES on a tree left dirty / mid-rebase. When the recover/integrate path threw mid-rebase, the tree was exactly that, so the `finally` release threw/aborted and did NOT land — leaving the slug in BOTH `work/advancing/<entry>.md` (orphaned lock) AND its lifecycle folder. (Live evidence: the incident's `origin/main` trail shows `advancing: lock` then `claim` then NO `advancing: release` commit.)

Fix (SHAPE — adapt to the reproduced cause): make the release land from whatever post-failure state the repro reveals, WITHOUT destroying the recoverable work. The release is ALREADY work-preserving + self-isolating (scratch `advancing-release/<entry>` branch, `originalRef` captured, `cleanup` checks `origRef` back out), so the fix is making it ROBUST to the post-failure worktree state, NOT rewriting it. Likely shapes, depending on the repro: (i) make the release tolerate a worktree that differs from HEAD due to the in-flight ledger/marker files (it never needs the dirty work — the CAS is on a scratch branch off `<arbiter>/main`), e.g. stash/clean only what blocks the checkout; (ii) if a rebase is genuinely still in progress, `git rebase --abort` (restores the kept tip) before the existing release runs. WHATEVER the shape: do NOT `git reset --hard` the work branch and do NOT discard commits — the kept work lives on the WORK BRANCH REF and must stay recoverable. The invariant to restore: **after any `advance` run (success OR failure), the slug is in exactly ONE lifecycle folder, holds NO orphaned advancing borrow, AND the kept work branch tip is intact + recoverable.**

While here, introduce the single path-construction seam **`advancingMarkerPath(entry)`** and route BOTH the acquire and release marker-path construction through it (replacing the inline `work/advancing/${entry}.md` at the two sites). This is the centralized helper the folder-taxonomy reorg will later repoint to relocate the marker (see `## Decisions`). KEEP `<type>-<slug>` in the lock BRANCH name (`advancing/<entry>` / `advancing-release/<entry>`) — it is load-bearing for cross-type/cross-repo branch-collision avoidance even when the marker filename later co-locates.

> IN-PLACE vs ISOLATED (verified — scope this precisely). The release runs in the LOCK's `cwd` (the advance checkout). For an IN-PLACE `advance` (the incident: `--propose --watch`, no `--isolated`) the recover/integrate rebase runs in that SAME `cwd`, so the throw leaves THAT dir dirty / mid-rebase and the release's dirty-guard trips / its `git checkout` of the scratch+orig refs refuses — the fix (become checkout-able via rebase-abort, then let the existing scratch-branch CAS run) targets exactly this. For an ISOLATED/worktree dispatch the build's dirty state is in a DIFFERENT job worktree than the release `cwd`, so the release `cwd` may already be clean; the fix must STILL be correct there (don't assume the dirt is local, and don't fail if it is elsewhere). Cover both in tests.
