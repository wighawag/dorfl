## What to build

Make the `advancing` lock release land even when the post-lock dispatch (recover / integrate / gate) THREW mid-operation, so a failed `advance` run never leaves an orphaned advancing marker and a one-slug-two-folder ledger.

`advance` takes the `advancing` CAS borrow (`work/advancing/<entry>.md`, `<entry>` = `<type>-<slug>`) and releases it in a `finally`. The release mechanism is ALREADY self-isolating and work-preserving: `runRelease` captures the current ref (`originalRef`), `releaseAttempt` does its marker-removal CAS on a SCRATCH `advancing-release/<entry>` branch cut from `<arbiter>/main` (never the work branch), and a `finally` `cleanup` does `git checkout <origRef>` + deletes the scratch branch — so the work branch tip is checked back out, never reset. The ACTUAL blocker is the guard at the TOP of `runRelease` that THROWS when the working tree/index is dirty ("commit/stash them before releasing") BEFORE any of that runs; and even past it, `git checkout` of the scratch/orig refs REFUSES on a tree left dirty / mid-rebase. When the recover/integrate path threw mid-rebase, the tree was exactly that, so the `finally` release threw/aborted and did NOT land — leaving the slug in BOTH `work/advancing/<entry>.md` (orphaned lock) AND its lifecycle folder. (Live evidence: the incident's `origin/main` trail shows `advancing: lock` then `claim` then NO `advancing: release` commit.)

Fix: make the release reach a CHECKOUT-ABLE state in the crash path so its existing scratch-branch CAS + `cleanup` can run — WITHOUT destroying the recoverable work. The kept work lives on the WORK BRANCH REF (and the in-progress rebase's pre-rebase HEAD), NOT in the dirty working tree, so the safe move is: ABORT any in-progress rebase (`git rebase --abort`, which restores the branch's pre-rebase tip) and/or clear the index so the tree is checkout-able — then let the EXISTING release run (it already operates on a scratch branch and restores `origRef`). Do NOT `git reset --hard` the work branch and do NOT discard commits: a `git rebase --abort` leaves the branch ref at its kept tip (recoverable), which is the whole point. The invariant to restore: **after any `advance` run (success OR failure), the slug is in exactly ONE lifecycle folder, holds NO orphaned advancing borrow, AND the kept work branch tip is intact + recoverable.**

While here, introduce the single path-construction seam **`advancingMarkerPath(entry)`** and route BOTH the acquire and release marker-path construction through it (replacing the inline `work/advancing/${entry}.md` at the two sites). This is the centralized helper the folder-taxonomy reorg will later repoint to relocate the marker (see `## Decisions`). KEEP `<type>-<slug>` in the lock BRANCH name (`advancing/<entry>` / `advancing-release/<entry>`) — it is load-bearing for cross-type/cross-repo branch-collision avoidance even when the marker filename later co-locates.

## Acceptance criteria

- [ ] After an `advance` run whose post-lock dispatch THROWS (e.g. a rebase conflict in the recover/integrate path), the `advancing` borrow is CLEARED on the arbiter (no orphaned `work/advancing/<entry>.md`) and the slug is in exactly ONE lifecycle folder. A throwaway-git fixture induces a throwing integrate under a held advancing lock and asserts the marker is gone + one-folder afterward — covers story 4.
- [ ] **The kept work is PRESERVED, not destroyed, by the crash-safe release.** A test asserts that after the throwing-dispatch release the work branch tip still points at the kept commit (recoverable) — the fix aborts the in-progress rebase (restoring the pre-rebase tip) / clears the index to become checkout-able, and NEVER `git reset --hard`s or discards the branch's commits. This is the load-bearing safety property (the recover path exists to keep stranded work recoverable).
- [ ] The release no longer fails merely because the worktree/index is dirty / a rebase is in progress: it becomes checkout-able first (rebase-abort / index-clear), then the EXISTING scratch-branch marker-removal CAS + `cleanup` run against fresh `main`, NEVER `--force`. A test pins the dirty/mid-rebase case releases cleanly AND restores the original branch (the existing `originalRef`/`cleanup` behaviour is preserved, just reachable now).
- [ ] The happy-path release is UNCHANGED (a successful `advance` still releases exactly as before) — no regression to the existing acquire/release tests.
- [ ] `advancingMarkerPath(entry)` is the SINGLE place the `work/advancing/<entry>.md` path is constructed; both acquire and release call it. `<type>-<slug>` remains in the lock/release BRANCH names. A test asserts the path is computed via the helper (and the branch name still carries the type-encoded entry).
- [ ] Tests use throwaway `--bare` `file://` arbiters + real clones (the existing advancing-lock test style); point any workspace at a temp dir; no network.
- [ ] `pnpm format` then `pnpm -r build && pnpm -r test && pnpm format:check` green.

> IN-PLACE vs ISOLATED (verified — scope this precisely). The release runs in the LOCK's `cwd` (the advance checkout). For an IN-PLACE `advance` (the incident: `--propose --watch`, no `--isolated`) the recover/integrate rebase runs in that SAME `cwd`, so the throw leaves THAT dir dirty / mid-rebase and the release's dirty-guard trips / its `git checkout` of the scratch+orig refs refuses — the fix (become checkout-able via rebase-abort, then let the existing scratch-branch CAS run) targets exactly this. For an ISOLATED/worktree dispatch the build's dirty state is in a DIFFERENT job worktree than the release `cwd`, so the release `cwd` may already be clean; the fix must STILL be correct there (don't assume the dirt is local, and don't fail if it is elsewhere). Cover both in tests.
