> Derives from `work/observations/intake-lone-slice-review-built-as-per-round-launches-not-in-context-loop.md` (2026-06-10) and the SETTLED mechanism in `work/prd/review.md` §"Shape 2" (resolved Q1) + `work/done/slicer-review-edit-loop.md`. The just-merged `intake-lone-slice-bounded-internal-review` (PR #62) shipped the bounded review as the RUNNER looping and launching a FRESH review agent PER ROUND (M=3 launches, N=1 pass each) — the exact INVERSION of the canonical model, where **N = review→edit passes INSIDE one agent launch** and **M = number of launches**. The built shape is behaviourally correct but wasteful + conceptually off: a fresh-context-per-round agent can only build on the prior round via disk (FORBIDDEN — no write before convergence) or via the prompt (what it does — making the 3 cold-start launches near-pointless). This slice reshapes the mechanism to ONE agent doing its own bounded in-context review-edit loop, keeping every shipped behaviour and the no-disk-before-converge invariant. It does NOT change the lone-slice path's OUTCOMES, just HOW the rounds run.

## What to build

In `dispatchSlice` (`packages/agent-runner/src/intake.ts`) — the `slice` branch ONLY — replace the per-round runner loop with a SINGLE bounded in-context review pass:

- **Collapse the seam from per-ROUND to per-REVIEW.** Today `LoneSliceReviewGate` is called once PER ROUND by `runLoneSliceReview`'s `for` loop, and `harnessLoneSliceReviewGate` does one `launchWithOptionalWatch` per call (distinct `sessionId: …-r${round}`). Change it so the gate is invoked ONCE: a single agent launch whose PROMPT instructs the agent to run its OWN bounded (≤3, hard-capped) review→edit→re-review loop in its single context, and EMIT a terminal result — either the converged (possibly edited) slice body, OR a non-converge verdict carrying the open question(s). The `for round in 1..3` loop in the runner GOES AWAY (the rounds move INSIDE the agent's reasoning); `runLoneSliceReview` becomes a thin "launch once, parse the terminal verdict, map to converge/non-converge" function.

  > NOTE — the removed per-round-launch loop is NOT worthless: it is the structural template for the **M (fresh-context) layer** of the settled M×N model (`prd/review.md` §"Shape 2"). REMOVE it from the intake path here (intake's rulings A/B mandate `M=1`, no M×N knob), but its shape is captured for possible later revival in `work/ideas/lone-slice-review-fresh-context-m-layer.md` (which also records the de-chaining it would need to be a COHERENT M loop). Do NOT preserve it as dead code in `intake.ts` to "keep the option" — the idea file is its home; delete it from the path cleanly.

- **The 3-round cap stays HARD-CODED (rulings A/B unchanged)** — but it is now an INSTRUCTION in the prompt ("do at most 3 internal review-edit passes, then stop") rather than a runner-side launch counter. Still a literal constant (keep `LONE_SLICE_REVIEW_MAX_ROUNDS = 3` and inject it into the prompt text); NOT configurable, NO flag, NOT optional. (Mirrors how the slicer edit loop's N is the in-context pass depth, not M launches.)
- **No disk write before convergence stays invariant.** The single launch returns the final body as TEXT (the terminal verdict's `edit`/body field); the runner writes `work/backlog/<slug>.md` ONCE, only on convergence, exactly as today. Nothing is written on the non-converge path.
- **The injectable seam stays — reshaped for testability.** Keep an injectable seam (`PerformIntakeOptions.reviewSlice`, mirror `IntakeDecider`) so tests drive a CANNED terminal verdict with no model/network — but it now represents the WHOLE bounded review (one call → terminal result), not one round. Tests inject the final disposition directly (converge-with-edit / non-converge-with-questions / launch-or-parse-failure).
- **Fold in the early-flip fix (nit #1 from PR #62).** Because the loop is now in-context, the agent can converge OR decide a blocking question has no thread answer and return NON-CONVERGE on the single launch — so the SLICE→ASK flip happens when the agent says so (early or at its internal cap), not only after a runner-side 3rd round. The terminal verdict expresses both outcomes directly. (This retires the "flips only at the cap" behaviour the PR #62 review flagged.)
- **The prompt:** the in-context loop was DOGFOOD-VALIDATED at the original slice's authoring (see the observation's "The dogfooding ALREADY validated the in-context loop" — a fresh agent ran the bounded ≤3-round self-review in ONE context, converged, and the round-2 findings materialised in the final artifact). Build the production prompt by reframing the EXISTING `buildLoneSliceReviewPrompt` lens content (per-slice well-formedness + DESTINATION check; SET/graph/overlap OFF, N=1) from "you are reviewing round R of 3" to **"run your OWN bounded review→edit→re-review loop, at most 3 internal passes, then emit a TERMINAL verdict — the converged (possibly edited) body, or a non-converge with the open question(s)."** Two framings from the dogfooded prompt must NOT be copied into production (they were dry-run-only): (1) do NOT tell the agent to TREAT the source AS IF an issue thread — the real intake reads an ACTUAL issue; pass the real drafted slice; (2) do NOT add a `REVIEW-NOTES` / round-trace reporting block — the in-code loop's rounds are internal/invisible (the dry-run only reported them so the human could see the loop ran). The dogfooded prompt is a SHAPE reference, never a verbatim source.

Keep the verdict/output CONVENTIONS (a fenced JSON `{verdict, findings, edit, questions}` parsed via the shared `extractJsonObjectSpan` / `parseLoneSliceReviewVerdict`) — only the SEMANTICS shift from "one round's verdict" to "the terminal verdict of the agent's internal loop". Do NOT import or call `runSliceReviewLoop`. Do NOT add a synthetic-source seam. Do NOT touch PRD / ASK / BOUNCE. ZERO new `IntakeRunOutcome`, ZERO new marker kind, ZERO config flag.

Asserted: a test pins the production gate invokes the harness EXACTLY once for one lone-slice review (not 3).
- [ ] CONVERGE: a terminal verdict with the (possibly edited) body → the slice is written to `work/backlog/<slug>.md` and integrated exactly as today; outcome `sliced`; the `slice created` completion comment is posted. The emitted body reflects the agent's edit. No `work/backlog/` write occurs before the terminal result (tested: no file written when the verdict is non-converge).
- [ ] NON-CONVERGE: a terminal verdict carrying open question(s) (the agent judged a blocking question unanswerable from the thread, whether at its 1st internal pass or its cap) → verdict flips `slice → ask`: outcome `asked`, NO slice written, NO integrate; ONE comment via the existing `dispatchComment`/`postIssueComment` path carrying BOTH the proposed slice DRAFT and the open question(s), stamped `markerKind: 'ask'`. Asserted at the stubbed seam. (This subsumes + retires the PR #62 "flips only at the 3rd round" behaviour: the flip is driven by the agent's terminal verdict, which can be reached early.)
- [ ] The review gate DEGRADES honestly: a launch/parse failure maps onto the existing `agent-failed` outcome (exit 1) via the dispatcher's try/catch — never a silent emit of the un-reviewed slice. A test pins this.
- [ ] Zero new `IntakeRunOutcome`, zero new marker kind, zero config flag (the diff adds none — verified by inspection). The non-converge path uses the EXISTING `asked` outcome + EXISTING `kind=ask` marker; the next run resumes via the already-built triage gate.
- [ ] `runSliceReviewLoop` is NOT imported or called from `intake.ts`; PRD / ASK / BOUNCE outcomes are untouched (only `dispatchSlice` / the `slice` branch + the lone-slice review seam change). Verified by inspection.
- [ ] Tests STUB the issue seam + the (reshaped, single-call) review seam (no network), mirroring the existing intake dispatcher tests; the review prompt's JUDGEMENT is NOT unit-tested (only the single-launch flow + the converge/flip dispatch), exactly as the decision prompt's is not.
- [ ] `pnpm -r build && pnpm -r test && pnpm -r format:check` green.
