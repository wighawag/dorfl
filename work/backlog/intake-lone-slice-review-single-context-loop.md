## What to build

In `dispatchSlice` (`packages/agent-runner/src/intake.ts`) вЂ” the `slice` branch ONLY вЂ” replace the per-round runner loop with a SINGLE bounded in-context review pass:

- **Collapse the seam from per-ROUND to per-REVIEW.** Today `LoneSliceReviewGate` is called once PER ROUND by `runLoneSliceReview`'s `for` loop, and `harnessLoneSliceReviewGate` does one `launchWithOptionalWatch` per call (distinct `sessionId: вЂ¦-r${round}`). Change it so the gate is invoked ONCE: a single agent launch whose PROMPT instructs the agent to run its OWN bounded (в‰¤3, hard-capped) reviewв†’editв†’re-review loop in its single context, and EMIT a terminal result вЂ” either the converged (possibly edited) slice body, OR a non-converge verdict carrying the open question(s). The `for round in 1..3` loop in the runner GOES AWAY (the rounds move INSIDE the agent's reasoning); `runLoneSliceReview` becomes a thin "launch once, parse the terminal verdict, map to converge/non-converge" function.

  > NOTE вЂ” the removed per-round-launch loop is NOT worthless: it is the structural template for the **M (fresh-context) layer** of the settled MГ—N model (`prd/review.md` В§"Shape 2"). Do NOT preserve it as dead code in `intake.ts` to "keep the option" вЂ” the idea file is its home; delete it from the path cleanly.

- **The 3-round cap stays HARD-CODED (rulings A/B unchanged)** вЂ” but it is now an INSTRUCTION in the prompt ("do at most 3 internal review-edit passes, then stop") rather than a runner-side launch counter. Still a literal constant (keep `LONE_SLICE_REVIEW_MAX_ROUNDS = 3` and inject it into the prompt text); NOT configurable, NO flag, NOT optional. (Mirrors how the slicer edit loop's N is the in-context pass depth, not M launches.)
- **No disk write before convergence stays invariant.** The single launch returns the final body as TEXT (the terminal verdict's `edit`/body field); the runner writes `work/backlog/<slug>.md` ONCE, only on convergence, exactly as today. Nothing is written on the non-converge path.
- **The injectable seam stays вЂ” reshaped for testability.** Keep an injectable seam (`PerformIntakeOptions.reviewSlice`, mirror `IntakeDecider`) so tests drive a CANNED terminal verdict with no model/network вЂ” but it now represents the WHOLE bounded review (one call в†’ terminal result), not one round. Tests inject the final disposition directly (converge-with-edit / non-converge-with-questions / launch-or-parse-failure).
- **Fold in the early-flip fix (nit #1 from PR #62).** Because the loop is now in-context, the agent can converge OR decide a blocking question has no thread answer and return NON-CONVERGE on the single launch вЂ” so the SLICEв†’ASK flip happens when the agent says so (early or at its internal cap), not only after a runner-side 3rd round. The terminal verdict expresses both outcomes directly. (This retires the "flips only at the cap" behaviour the PR #62 review flagged.)
- **The prompt:** the in-context loop was DOGFOOD-VALIDATED at the original slice's authoring (see the observation's "The dogfooding ALREADY validated the in-context loop" вЂ” a fresh agent ran the bounded в‰¤3-round self-review in ONE context, converged, and the round-2 findings materialised in the final artifact). Build the production prompt by reframing the EXISTING `buildLoneSliceReviewPrompt` lens content (per-slice well-formedness + DESTINATION check; SET/graph/overlap OFF, N=1) from "you are reviewing round R of 3" to **"run your OWN bounded reviewв†’editв†’re-review loop, at most 3 internal passes, then emit a TERMINAL verdict вЂ” the converged (possibly edited) body, or a non-converge with the open question(s)."** Two framings from the dogfooded prompt must NOT be copied into production (they were dry-run-only): (1) do NOT tell the agent to TREAT the source AS IF an issue thread вЂ” the real intake reads an ACTUAL issue; pass the real drafted slice; (2) do NOT add a `REVIEW-NOTES` / round-trace reporting block вЂ” the in-code loop's rounds are internal/invisible (the dry-run only reported them so the human could see the loop ran). The dogfooded prompt is a SHAPE reference, never a verbatim source.

Keep the verdict/output CONVENTIONS (a fenced JSON `{verdict, findings, edit, questions}` parsed via the shared `extractJsonObjectSpan` / `parseLoneSliceReviewVerdict`) вЂ” only the SEMANTICS shift from "one round's verdict" to "the terminal verdict of the agent's internal loop". Do NOT import or call `runSliceReviewLoop`. Do NOT add a synthetic-source seam. Do NOT touch PRD / ASK / BOUNCE. ZERO new `IntakeRunOutcome`, ZERO new marker kind, ZERO config flag.
