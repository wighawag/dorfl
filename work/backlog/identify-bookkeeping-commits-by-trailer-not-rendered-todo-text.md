## What to build

The runner drops its own `route to needs-attention` MOVE-ONLY bookkeeping commits during two rebases (`rebaseDroppingBookkeepingMoves` in `src/drop-bookkeeping-rebase.ts`, called by the integration-recovery rebase in `integration-core.ts` and the onboard continue-rebase in `continue-branch.ts`). Today it identifies which commits to drop with a `sed` `GIT_SEQUENCE_EDITOR` that pattern-matches git's RENDERED `git-rebase-todo` line (`/^pick <sha> \(# \)\{0,1\}chore(<slug>): route to needs-attention/d`).

That rendered-todo text is a human-facing PRESENTATION detail, not a stable interface, and it has drifted across git versions: the `# ` instruction prefix (default ~2.31), abbreviated command names (`rebase.abbreviateCommands`), hash-abbreviation length, and the full-subject `# %s` `instructionFormat` that became the DEFAULT in git 2.54. When the rendering changes, the regex silently stops matching, the bookkeeping commit is NOT dropped, it REPLAYS onto a main that no longer has that ledger state, and the rebase conflicts — the agent self-conflicting with the runner's own protocol bookkeeping.

VERIFIED: this is the live CI failure. `complete-from-needs-attention.test.ts` (the recovery `complete` path, which rebases dropping the historical `in-progress → needs-attention` move) passes on git 2.47/2.53 locally but FAILS on the GitHub `ubuntu-latest` runner (git 2.54.0) with `rebase-conflict` instead of `completed` / a missing done-move / `ENOENT … work/done/<slug>.md`. Reproduced the version split locally: passes under git 2.53, the 2.54 default `instructionFormat` is `# %s` (confirmed in 2.54 `sequencer.c` `sequencer_make_script`). The earlier `continue-rebase-auto-resolves-protocol-bookkeeping-conflicts` slice (now in `done/`) hardened the SAME helper against the `# ` prefix but is still coupled to the rendered line and so still breaks on the next rendering change.

THE FIX (decouple identification from rendering). Two halves:

### Half 1 (PRODUCER, the root) — stamp a durable marker when the commit is created

Where the runner authors the move-only commit (`needs-attention.ts` — there are TWO author sites, and they commit by DIFFERENT mechanisms: the in-worktree `routeToNeedsAttention` ~L378 uses `git commit -m <commitMessage>`, while the tree-less ledger-move ~L752 builds the commit via `runTreelessLedgerMove` → `git commit-tree` with the message passed as a STRING — there is no worktree `commit -m` at the second site, so do not go looking for one). Both produce the subject `chore(<slug>): route to needs-attention; <reason>`. Add a structured git TRAILER to that commit message at BOTH sites (at the tree-less site, append it to the message string handed to `commit-tree`), e.g.:

```
chore(<slug>): route to needs-attention; <reason>

Agent-Runner-Bookkeeping: route-to-needs-attention
```

The trailer is the EXPLICIT, intentional mark the producer puts on the commit — nothing is inferred from prose. It is part of the commit OBJECT, so it travels with the kept `work/<slug>` branch to the arbiter and to any other machine automatically (this is why a marker, not an in-memory sha hand-off, is required: the onboard continue-rebase runs in a FRESH process on a possibly DIFFERENT machine than the one that created the commit, so there is no in-memory value to pass — the identifier must be RECORDED on the commit).

### Half 2 (CONSUMER) — compute the drop-set from plumbing, regenerate the todo ourselves

In `drop-bookkeeping-rebase.ts`, identify the commits to drop by reading the REAL commit data via plumbing — read the TRAILER over `base..HEAD` however is portable (the requirement is identification-by-recorded-trailer, NOT a specific git atom; `git log --format=%H%x00%(trailers:key=Agent-Runner-Bookkeeping)` works, as does `git interpret-trailers --parse` per commit — pick what is robust on the repo's git, and do NOT reintroduce a version dependency in the very fix for one). Anchor so ONLY this slug's route-to-needs-attention commits match and NEVER a completed-state move (`→done` / `slicing → prd-sliced`). Then drive the non-interactive `git rebase -i` with a `GIT_SEQUENCE_EDITOR` that REWRITES the todo from OUR computed ordered sha list (one `pick <fullsha>` line per kept commit, dropped shas omitted; `noop` if the kept set is empty) — never reading or matching git's own rendered todo text. `pick <fullsha>` is the canonical version-stable instruction form git parses identically regardless of how it renders. (Equivalent plumbing-only mechanisms — e.g. an explicit `--onto` range surgery that excludes the dropped shas — are acceptable if they too avoid matching rendered todo text; the invariant is "identify by recorded trailer, drive by sha, never by rendered line".)

THE HARD INVARIANT is unchanged and must be preserved: drop ONLY runner-authored `route to needs-attention` move-only commits for THIS slug; NEVER drop a completed-state move; a genuine code conflict still present after the drop still aborts → needs-attention (the caller's existing abort path governs). Behaviour must be IDENTICAL across git versions.

BACKWARD COMPATIBILITY: a kept `work/<slug>` branch created BEFORE this change carries an OLD move-only commit with NO trailer. Decide and record (see Decisions guidance in the Prompt) whether the consumer also falls back to the subject-anchored match for un-trailered commits (belt-and-suspenders for in-flight branches) or whether the subject match is removed entirely. A reasonable default: keep recognising the legacy subject form AS WELL AS the trailer for one transition, since both are exact and slug-anchored; but if kept, it must be matched via the REAL subject (`%s` plumbing), NOT via the rendered todo line. NOTE there are currently LIVE pre-existing kept `work/<slug>` branches on the arbiter (e.g. `work/do-lost-on-in-progress-resume-hint`, `work/slice-stale-lease-retry-all-push-sites-and-treeless-surface`); if any carry an un-trailered route-to-needs-attention commit and you DROP the legacy fallback, a continue/recovery on THOSE branches would silently stop dropping — make the decision with that exposure in mind.

PROTOCOL DOCS: the move-only commit's message shape is protocol bookkeeping. Adding the trailer changes that shape, so update the SOURCE protocol docs (`skills/setup/protocol/*` — e.g. `WORK-CONTRACT.md` where the route-to-needs-attention commit is described) and mirror byte-identically into `work/protocol/*` (see this repo's AGENTS.md: edit the SOURCE, never the propagated copy alone; `diff -r skills/setup/protocol work/protocol` stays clean).
