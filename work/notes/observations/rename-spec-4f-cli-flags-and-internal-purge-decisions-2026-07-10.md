## Decisions

### D1 — `scan --json` output field `specs[]` → `specs[]` (a MACHINE OUTPUT CONTRACT change)

The task's local-var list says rename `specs`→`specs` "file-local, rename freely" and names `scan.ts`/`mirror-pool-scan.ts`. But `RepoReport.specs` (scan.ts:172) is the **`scan --json` OUTPUT field**, `JSON.stringify`'d and read by the CI propose-matrix `jq` as `.repos[].specs[]?` / `.cwd.repo.specs[]?` (advance-lifecycle-template.ts + docs/ci/advance-loop.yml.template + the compiled `.github/workflows/advance-lifecycle.yml`). So renaming it is NOT a mere local-var rename — it changes a machine-consumed output schema.

I renamed it to `specs[]` anyway, because (a) the task's acceptance grep flags the bare `specs` struct-field identifier (it excludes `prd:` but not `specs`), and the contract task's leak scan targets internal identifiers too (batch-4f retask: "scan widened to internal names"), and (b) I updated the coupled SOURCE consumers in lockstep: the two CI templates' `jq` (`.specs[]` → `.specs[]`) + `docs/ci/advance-loop.yml.template` + the coupled template tests. The `"spec:" + .slug` MATRIX LEGS were KEPT (the task grep excludes `prd:`; `prd:<slug>` is still an accepted CLI alias the contract task removes later).

**Touches / follow-up:** the compiled `.github/workflows/advance-lifecycle.yml` still reads `.specs[]` and must be REGENERATED (`install-ci`) by a human — otherwise propose-mode CI silently enumerates ZERO taskable specs (a `jq` reading a now-missing field returns empty, no loud error). This is the SAME compiled-workflow-regen follow-up the CLI-flag clean break already needs (D2). See "Manual follow-up" below.
