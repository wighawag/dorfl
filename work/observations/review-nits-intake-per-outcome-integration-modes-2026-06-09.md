## Non-blocking review findings

The PR/code review gate (Gate 2) APPROVED 'intake-per-outcome-integration-modes' but raised the following non-blocking findings (nits). They do not block integration; this is their durable home for triage — promote-to-slice / keep / delete.

- Ratify: the per-type usage error is REWORDED ('--merge-prd and --propose-prd are mutually exclusive; pass at most one.') rather than surfacing the reused aggregate message. The slice said to reuse `integrationFromFlags`'s error message, but also asked for a 'clear message' naming the granular pair. The agent reused the LOGIC but re-worded the MESSAGE per type. This is a sensible UX choice (a user who typed `--merge-prd --propose-prd` should see those flags named, not '--merge and --propose'), but it is a decision not literally spelled in the slice. Ratify or prefer the verbatim reused message? (src/intake.ts `granularFromFlags` catches the reused throw and rethrows with the per-type flag names. The aggregate `--merge --propose` path keeps the original message verbatim. Tested in intake-integration-modes.test.ts (three separate mutual-exclusion assertions).)
- Ratify a cross-cutting CLI refactor: the `intake` command NO LONGER injects the flag-derived mode into `resolveRepoConfig` (the old `...(flagMode ? {integration: flagMode} : {})` block was removed). Instead it passes the per-repo/global-resolved `config.integration` as the `defaultMode` fallback to `resolveIntakeIntegrationModes`, and the aggregate `--merge`/`--propose` flags are now resolved INSIDE that function via `aggregate ?? defaultMode`. The net precedence (granular > aggregate flag > per-repo > global > default propose) is preserved and the resolver's docstring/tests assert it, but the LOCUS of aggregate-flag precedence moved from config-resolution into the intake resolver — a design choice specific to `intake` (other commands still inject flagMode into resolveRepoConfig). Confirm this is the intended shape and not an accidental behavioural divergence for any per-repo `integration: merge` repo running bare `intake`.
