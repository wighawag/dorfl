## Non-blocking review findings

The PR/code review gate (Gate 2) APPROVED 'intake-tracer-slice-outcome' but raised the following non-blocking findings (nits). They do not block integration; this is their durable home for triage — promote-to-slice / keep / delete.

- The intake commit subject and propose-PR title degrade to the generic 'complete work slice' / 'feat(<slug>)' instead of carrying the drafted slice title — is that acceptable, or should the title be threaded through? (dispatchSlice() sets lifecycle.titlePath = join(cwd, 'work/backlog/<slug>.md') and its doc-comment claims 'The emitted slice IS the title source — read it from the path the stage writes (the file persists).' But integration-core.ts reads the title (lines 509-511: defaultSummary/readSliceTitle/synthesiseProposeTitle) BEFORE lifecycle.stage() writes that file (line 524). For the slicing path titlePath is an already-existing held PRD, so it works; for intake the output file does not exist yet at read time, so readSliceTitle returns undefined and the commit subject falls back to 'complete work slice' and the PR title to 'feat(<slug>)'. The slice FILE's frontmatter title is correct (renderBacklogSlice writes 'title: ' + sliceTitle); only the commit/PR title is lost. No test asserts the commit subject or PR title, so it passed the green gate. Easy fix: write the file in dispatchSlice before performIntegration, or pass the title via the `message` option / lifecycle. Cosmetic-only, hence non-blocking.)
- Ratify the deliberate decision: the PRODUCTION intake decision path is a stub that always throws — a real `intake <N>` invocation cannot produce a slice today; only the test-injected `decide` path works. Intended deferral to the next slice? (intake.ts runDecision(): when no `decide` is injected, the harness branch launches then unconditionally throws 'the production intake decision parse is not wired yet (see intake-decision-prompt-and-four-outcome-dispatch); inject `decide` to drive the dispatcher.' This is consistent with the slice's framing ('test the DISPATCH, not the model'; the full prompt asset + verdict parse are the next slice) and its 'Done' definition (proven via a STUBBED verdict). But it means the shipped `intake` CLI command, run for real by a human, errors rather than emitting a slice — PRD US #6's 'run LOCALLY one-shot' is only end-to-end-real once intake-decision-prompt-and-four-outcome-dispatch lands. Reads as intended; flagging for explicit human ratification since it is a user-visible behaviour of the newly-shipped command.)
- The gate-free test builds a config with allowAgents:false/autoSlice:false but then `void config`s it (never passes it to performIntake) — is the assertion meaningful, or tautological? (intake.test.ts 'is GATE-FREE' case: since performIntake takes no autonomy config at all, the run cannot consult those gates and the test's success is structurally guaranteed (it documents/locks the gate-free property rather than proving a branch). This is the correct shape (intake genuinely never reads autoSlice/autoBuild) and matches the slice AC wording, but a reader should know the test asserts the ABSENCE of a config knob, not a gate being bypassed. Minor.)
