## Non-blocking review findings

The PR/code review gate (Gate 2) APPROVED 'failure-cause-classification-model-vs-git-vs-agent' but raised the following non-blocking findings (nits). They do not block integration; this is their durable home for triage — promote-to-slice / keep / delete.

- Ratify: failure-cause classification is purely lexical (regex over the surfaced error message) with an ordered precedence (config-error before transient-infra). This is best-effort by design, but it is brittle to wording drift in what the harness/git surfaces. Is the lexical-only approach acceptable as the long-term mechanism, or should config-error eventually key off a typed/structured error rather than the 'wiring bug' phrase? (src/failure-cause.ts CONFIG_ERROR_SIGNATURES + TRANSIENT_INFRA_SIGNATURES; classifyFailureCause matches on phrases the catch sites hand it as err.message only. Slice explicitly scopes this as best-effort + conservative, so this is a ratification, not a defect.)
- Ratify the breadth of the transient-infra signature set: it matches bare HTTP status codes /\b(429|500|502|503|504)\b/ and generic phrases (/timed? ?out/, /rate.?limit/, /network error/). An agent/build error whose output incidentally contains '500' or 'timeout' could be mis-labelled transient-infra (whose recovery is 'retry the same work'). Acceptable false-positive risk? (src/failure-cause.ts TRANSIENT_INFRA_SIGNATURES. Mitigated by the conservative-default framing and the cheap recovery (retrying fine work), but a human should confirm the broad lexical net is intended.)
- Ratify: the runOnce summary COUNTER counts transient-infra/config-error under 'failed' but not under 'needsAttention', mirroring the pre-existing treatment of agent-failed. These items DO route to needs-attention/ (tests assert it), so they are absent from the needsAttention tally. Keep the inherited behaviour, or fix agent-failed + the two new causes together so the needsAttention count reflects them? (src/run.ts ~352-376: needsAttention counter omits agent-failed (and now its two refinements); failed counter includes all three. Verified against HEAD that agent-failed was already excluded from needsAttention — this is consistent, not newly introduced.)
- Coherence nit (pre-existing): the new CONTEXT.md glossary entry says 'a red gate is the existing gate-failed', but run surfaces a red gate as ItemStatus 'tests-failed' (the core uses 'gate-failed'). The glossary the slice adds doesn't acknowledge run's tests-failed alias. Worth a one-line note so the run/core naming asymmetry is pinned rather than silently papered over? (CONTEXT.md failure-cause entry vs run.ts:152 'tests-failed' and integration-core.ts:58 'gate-failed'. Asymmetry predates this slice; low impact.)
- Ratify the user-visible route-message wording change in do: 'Agent failed building <slug> (...)' became 'Agent run failed building <slug> [<cause>] (...)'. The structured outcome and the historical 'agent failed:' reason-prefix are preserved, so reason-matching tests don't regress, but the human-facing note string changed.
