## Non-blocking review findings

The PR/code review gate (Gate 2) APPROVED 'advance-loop-driver-registry-set-job-worktrees' but raised the
following non-blocking findings (nits). They do not block integration; this
is their durable home for triage — promote-to-slice / keep / delete.

- Ratify: the agent narrowed WITHIN-mirror concurrency to serial (`maxParallel:1, perRepoMax:1` passed into each `advanceOnce`), so a mirror's pool drains one tick at a time even though `advanceOnce` itself supports per-repo concurrency. Acceptance criterion 5 asks for race-correctness 'across mirrors AND within a mirror's pool'; the implementation keeps cross-mirror concurrency genuine but serialises within a mirror. Is this acceptable for the slice, or was within-pool parallelism expected here?
  (advance-loop-driver.ts: the inner `worker` passes `maxParallel:1, perRepoMax:1` to `advanceOnce`. The agent documents the WHY thoroughly (the `advancing` borrow's CAS + the tree-less surface/triage/apply rungs commit in the shared per-mirror `context.cwd`, and the borrow is race-correct only per distinct clone; two contenders sharing one checkout would corrupt HEAD/index, the same reason `run` serialises the per-repo claim). The borrow race-test still proves correctness via two distinct-clone batches. This is sound and honestly documented, but it is a real narrowing vs the criterion's literal 'within a mirror's pool' wording, with per-item-cwd isolation deferred to a follow-up. Worth an explicit ratify since `run-uses-advance-tick` will inherit this serialization characteristic.)
- Ratify: the agent changed the EXISTING `performDoRemote` claim-dir naming from a fixed `<repo>__remote` to a per-arg `<repo>__<sanitised-arg>` key. This is a cross-slice interaction (it alters `do --remote`'s on-disk claim-dir layout for every caller, not just the new advance driver). Is the per-arg keying ratified?
  (do.ts performDoRemote: `const claimKey = options.arg.replace(/[^a-zA-Z0-9._-]/g, '_')`. The reasoning is sound: the single-shot `do --remote` caller never raced one mirror, but the registry-set batch's per-mirror concurrency would race a shared claim dir, so keying per-arg (matching `run`'s `claimAgainstRepo` per-job-clone pattern) is required for correctness and is a pure hardening for the prior caller. No test asserts the literal old `__remote` path, so nothing breaks. Flagging because it is an unspecified change to another command's behaviour that the slice did not call out.)
- Ratify: `mirrorOriginUrl` falls back to `file://<mirrorPath>` when `git remote get-url origin` is unreadable, so a malformed/origin-less mirror resolves to SOMETHING addressable rather than throwing mid-batch. Is silently degrading to a `file://` arbiter URL the desired behaviour, or should an unreadable origin surface as a stuck/needs-attention signal for that mirror instead?
  (advance-loop-driver.ts mirrorOriginUrl(): the fallback keeps one bad mirror from aborting the registry sweep, which aligns with the 'one mirror never aborts the batch' contract. But a `file://<bare-mirror-path>` arbiter is unlikely to be a working push/integrate target, so the build against it will probably fail downstream rather than be flagged as a config problem. Low-stakes (it only affects misconfigured mirrors) and easily reversible, hence non-blocking, but worth a human's eye on whether a degraded mirror should be reported rather than silently re-pointed at its own bare path.)
- Minor doc inaccuracy (no behaviour impact): the `advanceRegistrySet` JSDoc says the advance build rung reaches `performIntegration` 'via `performDo` -> `performDoRemote`', but the injected `doDriver` REPLACES `performDo` entirely — the path is `doDriver` (`jobWorktreeDoDriver`) -> `performDoRemote`, never through `performDo`. Worth correcting for the next reader.
  (advance-loop-driver.ts class JSDoc on advanceRegistrySet vs advance.ts orchestrateDo (`const driver = context.doDriver ?? performDo`). The `jobWorktreeDoDriver` JSDoc in do.ts is accurate ('re-routes the orchestration onto performDoRemote'); only the driver-class comment overstates a `performDo` hop that does not happen. Purely cosmetic — the code is correct.)
