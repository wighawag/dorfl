## Non-blocking review findings

The PR/code review gate (Gate 2) APPROVED 'intake-decision-prompt-and-four-outcome-dispatch' but raised the following non-blocking findings (nits). They do not block integration; this is their durable home for triage — promote-to-slice / keep / delete.

- Ratify the IntakeRunOutcome literal rename: ask -> asked and bounce -> bounced, while prd stays prd (and sliced was already past-tense). Is the slight noun/verb asymmetry (prd vs asked/bounced/sliced) acceptable, or should prd become e.g. 'prd-emitted' for symmetry? (src/intake.ts IntakeRunOutcome union. The rename distinguishes the run's terminal STATUS from the decision-table VERDICT noun. It is fully internal — cli.ts consumes result.message/exitCode generically and no other module compares these literals — so nothing breaks either way. Flagged only for naming coherence.)
- Ratify the new user-visible default comment text for empty ask/bounce drafts ('Could you clarify issue #N so it can be acted on?' / 'This issue looks like multiple unrelated concerns — please file separate issues…'). Is posting a generic fallback the right behaviour, or should an empty drafted message be treated as an agent error instead? (src/intake.ts dispatchComment fallbacks. A thin fallback keeps the comment non-empty if the agent left question/bounceMessage blank. A test covers it. The alternative (refuse on empty) would surface a prompt regression louder rather than papering over it, but the fallback is the more degrade-cleanly choice.)
- Ratify the renderPrd default scaffold body emitted when the agent drafts no PRD body (a Problem Statement / Solution / User Stories stub referencing issue #N). Is emitting a placeholder PRD that 'do prd:' can later slice the right default, vs refusing the prd outcome when prdBody is empty? (src/intake.ts renderPrd. Keeps the emitted file a structurally valid, sliceable PRD. The scaffold says '(to be detailed; this PRD needs slicing via do prd:)', so it is honest about being a stub, but a human reviewer of the resulting PR will see a near-empty PRD.)
- Ratify the new IntakeResult.commented?: boolean field (true iff a comment was posted on the ask/bounce branches). Is this the field downstream/CI should read to know a comment landed? (src/intake.ts IntakeResult. New optional result field, set from postIssueComment's posted flag (advisory — degrades to false without throwing if gh is missing/unauthenticated). User-visible on the result object; not yet consumed by any CLI/CI code (that is runner-in-ci's).)
- Ratify the issue: parse guard rejecting 0 / negative / non-integer as undefined (treated as absent, not a parse error). Is silently dropping a malformed issue: value the right degrade, given a real issue number is always a positive integer? (src/frontmatter.ts: result.issue = rawValue !== '' && Number.isInteger(n) && n > 0 ? n : undefined. Consistent with the undefined-when-absent convention of the other axes, and tested for 'not-a-number'/'0'/'-3'. A malformed link therefore reads as no link rather than crashing the parser.)
