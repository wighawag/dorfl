# In-flight OLD-location `.agent-runner-job.json` present while building its own relocation slice

2026-06-14 — While building `job-record-out-of-committed-tree` (which relocates the per-job record to a sibling OUTSIDE the worktree and REMOVES the `.gitignore` entry), the runner that dispatched THIS job is itself an OLD binary: it wrote the live record at the OLD in-tree path `<work-id>/.agent-runner-job.json` (state `running`, harness adapter `null`). With the gitignore entry now removed, the runner's own broad `git add -A` completion commit would sweep this stray untracked record into the work branch — the very leak this slice closes — because the dispatching binary predates the fix.

This is benign/transient (the next runner build carries the relocation; the read-fallback keeps such old-location records discoverable) and self-resolving once agent-runner is rebuilt from this branch. But it is the textbook "in-flight old-location record" the slice's migration step anticipates, observed live during its own self-hosting build. Left the runner's live record untouched (the runner owns its git-state transitions, AGENTS.md); flagging so whoever integrates this slice can confirm the record is not committed onto the work branch.
