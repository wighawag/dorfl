## Problem Statement

Three related gaps sit on top of an integration engine that is ALREADY most of the way to the goal, but whose doctrine is unwritten, whose CI shape lags the engine, and whose propose path has a residual blind spot.

The shared insight underneath all three: **a clean `git` merge AND a human-approved diff both validate a change in the context it was AUTHORED, never the context it will LIVE.** Neither `git`'s clean-merge exit code nor a human reading a PR diff catches the merge that applies cleanly but is semantically broken (task A renames a symbol, task B adds a caller of the old name; both rebase clean; the merged tree fails to build). `blockedBy` only guards conflicts the author PREDICTED; semantic coupling is exactly the class you do not predict. The only proof a change is correct in the tree it lands in is **re-running acceptance on the post-rebase tree**.

1. **The land-time re-verify doctrine is unwritten, so it reads as accidental, not load-bearing.** The engine already does it (`freshWorktreeGate`, ON by default, gates the work branch REBASED onto current `<arbiter>/main` — the would-be-merged tree — for `verify` AND review), and a moved-`main` mid-push re-rebases + re-gates via the `mergeRetries` CAS loop. But the protocol docs (`WORK-CONTRACT.md`, `CLAIM-PROTOCOL.md`) never state the invariant, so "done/green" reads as the BUILD-time gate, and a future change could silently regress to trusting a clean rebase. The mode-agnostic primitive (`land = rebase onto current main + re-run verify + advance; a moved-main / lost-CAS INVALIDATES a prior green and re-arms verify`) is implemented but not named.

2. **The CI merge shape forbids parallelism the engine now makes safe.** `docs/ci/README.md` still mandates a SINGLE SEQUENTIAL merge job, justified by "parallel merge jobs would thrash the main-CAS." That justification predates `integrateLock` (serialises ONLY the land tail, keeping build/gate/review concurrent) + `mergeRetries` (turns a non-fast-forward push into re-rebase + re-gate + retry, never a thrash, never a `--force`). The `run` fleet ALREADY gets ultimate-parallelism-for-merge (parallel build, serialised land). CI does not — it leaves merge throughput on the table for a reason that no longer holds. Caveat: `integrateLock` is IN-PROCESS only; across separate CI jobs the CAS retry loop is the only cross-job serialiser, and its cap (`DEFAULT_MERGE_RETRIES = 5`) was sized for in-process siblings, not a wide matrix.

3. **Propose mode's re-verify covers the PR-PUSH tip, not the PR-MERGE-time tip.** The runner gates the rebased tip when IT integrates; for merge mode that moment IS the land (airtight). For propose mode the runner's job ends at "push branch + open PR" — a human (or auto-merge) clicks merge LATER, after `main` may have moved again, in a window the runner cannot re-gate because it is not the one merging. So propose's "verify ran on the merged tree" guarantee holds at push time, not merge time, and a human reviewing the diff has the same stale-base blind spot as a clean `git` merge.

The cost: a correct, hard-won safety property is undocumented and fragile; CI throughput is needlessly capped; and the DEFAULT integration mode (`propose`) has an unclosed drift window.

## Solution

Name the primitive once, then realise it as a git-alone FLOOR with a host-accelerated CEILING. Per the project steer: **git alone defines the correctness floor (it must be safe with nothing but `git push` + ref CAS against a bare `--bare` arbiter and `NoneProvider`); a capable host can only RAISE the ceiling, never be required for safety. GitHub is the benchmark — if the design degrades gracefully from GitHub down to bare git, any middling host (GitLab, Gitea, plain SSH) slots in between.**

- **One mode-agnostic landing primitive, written into the protocol.** Document the invariant the engine already implements: every land = fetch current `main` → rebase → re-run `verify` (and review) on the rebased tree → advance; a lost CAS / moved-`main` INVALIDATES any prior green and re-arms the gate. Merge mode runs it inline at the sequential/serialised land; propose mode runs it at the human checkpoint. Human review is ADDITIVE (intent/design/security), NEVER a substitute for the re-verify. Capture this as durable rationale in an ADR and as an invariant line in `WORK-CONTRACT.md` / `CLAIM-PROTOCOL.md` (edit `skills/setup/protocol/` SOURCE + mirror into `work/protocol/`, byte-identical per this repo's protocol-docs rule).

- **CI parallel-merge shape, sized for cross-job landing.** Update `docs/ci/README.md` (and the validated `advance-ci-template.ts`) so merge mode CAN fan out, with the cross-job land serialised by whichever mechanism the open questions resolve (a scaled CAS retry cap, and/or a cross-job concurrency group / ref-lock so losers QUEUE rather than bounce to needs-attention). Record explicitly: across runners the CAS loop IS the queue; within a runner the in-memory `integrateLock` is the optimisation.

- **Propose PR-merge-time safety, two-tier.** Floor (git-alone / bare host): the gap closes only when the RUNNER performs the merge (its `freshWorktreeGate` then covers the merge-time tip); a human-merged PR on a bare arbiter is gated at push time only — documented as a known limitation with the mitigation "let the runner merge" or "re-run verify after rebase before merging." Ceiling (GitHub benchmark): branch protection "require branches up to date before merging" + a required `verify` status check forces a rebase + re-verify against current `main` before the merge button works (tier 1); a merge queue (`merge_group` trigger) does speculative-rebase composition checking and removes rebase churn (tier 2). A capable-but-not-GitHub host slots between by whatever subset it supports.

From the user's perspective: the safety property is now a stated, testable contract rather than an emergent accident; merge mode in CI is as parallel as the engine allows without ever landing a semantically-broken merge; and a propose PR cannot be merged stale without a re-verify wherever the host (or the runner-as-merger) can enforce it, with an honest documented floor where it cannot.

## User Stories

1. As a maintainer, I want the protocol to STATE the land-time re-verify invariant (rebase onto current main → re-run verify → advance; a moved-main invalidates a prior green), so the safety the engine already provides is a named contract, not an accident a refactor can silently drop.
2. As a maintainer, I want ONE mode-agnostic landing primitive in the docs, with merge and propose described as two FRONTENDS to it (runner-inline vs human-checkpoint), so propose-vs-merge is understood as "is a human approval required before the land," not "is the build parallel" or "is the tree re-verified."
3. As a maintainer, I want it written that human review is ADDITIVE and never replaces the re-verify, so nobody treats "a human approved the PR" as proof the merged tree builds.
4. As a maintainer running CI, I want merge mode to fan out in parallel (parallel build/gate/review, serialised land), so merge throughput matches the `run` fleet's instead of being capped by a stale single-sequential-job rule.
5. As the runner across separate CI jobs, I want concurrent merge landings to QUEUE deterministically (not bounce to needs-attention under a burst), since the in-memory `integrateLock` does not span jobs and only the CAS loop does — so the cross-job serialiser is sized/shaped for the matrix width on purpose. (SEE OPEN QUESTION 1.)
6. As `docs/ci/README.md`, I want my "parallel merge thrashes the CAS" claim corrected to reflect `integrateLock` + `mergeRetries`, so the doc stops contradicting the engine.
7. As a maintainer on GitHub, I want a propose PR to be UN-MERGEABLE while stale: branch protection "require up to date" + a required `verify` check forces a rebase + re-verify against current `main` before the merge button works, so the PR-merge-time drift window closes (tier 1).
8. As a maintainer on GitHub at scale, I want an optional merge queue (`merge_group` trigger on the verify workflow) that speculatively rebases + verifies + merges in order, so two individually-green PRs that break TOGETHER are caught and the rebase churn of tier 1 is removed (tier 2). (SEE OPEN QUESTION 3.)
9. As a maintainer on a bare / no-host / less-capable-host repo, I want the runner ITSELF to close the merge-time gap (not just document it): the advance loop SURFACES unmerged `work/*` branches / open PRs as MERGE-QUESTIONS, my ANSWER is the approval, and the apply step LANDS an answered-merge through the re-verify primitive — so the gap closes by the runner being the merger, with or without a host PR surface. (RESOLVES OPEN QUESTION 2 — see the surface/apply finding + the dedicated stories below; the human-merges-a-stale-PR limitation only remains if a maintainer DELIBERATELY merges outside the runner.)
10. As a maintainer, I want the host tiers framed as RAISING a ceiling over a safe git-alone floor (never as a safety prerequisite), so the tool stays honest on bare git and merely BETTER on GitHub — GitHub as the benchmark that proves the gradient.
11. As a maintainer adopting the GitHub ceiling, I want `install-ci` to AUTO-CONFIGURE Tier 1 (the required `verify` check + `strict: true` = require-up-to-date, via one `gh api` branch-protection / ruleset call) WHEN it is run with an admin-scoped credential, and to fall back to printing the exact ready-to-run command + manual instructions when it has only the ambient `GITHUB_TOKEN` — so the gate is reliably present where it can be, and never silently skipped where it cannot. (RESOLVED — see the verified finding + Implementation Decisions; the same call also creates the Tier-2 `merge_queue` ruleset rule.)
12. As a maintainer, I want a test that proves the land-time re-verify actually catches a clean-rebase-but-semantically-broken merge (A renames a symbol, B adds a caller; both rebase clean; the merged tree fails `verify`) — so the invariant is enforced by a test, not just asserted in prose.
13. As a maintainer, I want a test that proves concurrent merge landings serialise correctly (one lands, the other re-rebases + re-gates + lands or routes; never both-land-broken by timing; never a `--force` to main), at both in-process (`integrateLock`) and cross-job (CAS-loop) levels.
14. As the advance loop, I want a SECOND surfacer that enumerates unmerged `work/*` branches (reachability against `main` — the git-alone floor) and, where a host exists, open PRs (`gh pr list` — the ceiling rendering), and surface each as a MERGE-QUESTION into the SAME sidecar shape with a `merge|hold|drop` disposition — so propose-mode landing flows through the existing surface->answer->apply machinery, not a bespoke verb. (The judgement `surface-questions` skill is untouched; this is a state-sourced surfacer feeding the same sidecar.)
15. As the apply step, I want an answered `merge` disposition to invoke the LAND primitive (rebase onto current main -> re-run verify on the rebased tip -> advance), exactly as an answered `promote-slice`/`dropped` disposition triggers its `git mv` today — so "answer drives a runner action" reuses the existing disposition-dispatch pattern, now extended to the land action.
16. As a maintainer, I want an answered-merge to be a CONDITIONAL request, not a guarantee: if `main` moved between my answer and apply and the rebased tip FAILS `verify`, apply must REFUSE to merge (route to needs-attention or re-surface), never land a clean-rebase-but-broken tree — so my approval means "merge IF it still verifies on current main," and the clean-rebase-but-broken case is caught at the last moment. (Sub-decision in OPEN QUESTION 6: does a CHANGED merge-base re-surface even on a GREEN re-verify?)
17. As a maintainer, I want MERGE-QUESTIONS gated by their OWN config axis, SEPARATE from `observationTriage` and defaulting HIGHER, because "should this built work merge?" is materially more consequential than "is this observation worth promoting?" — an observation question defaults `off` (safely ignorable), but a merge-question silently dropped means finished, pushed work never lands. So merge-questions must NOT ride the observation gate. (SEE OPEN QUESTION 7 for the exact gate name + default.)

## Testing Decisions

- **The load-bearing test is the clean-rebase-but-broken-merge.** Two work branches, disjoint files, no textual conflict: A renames a symbol, B adds a caller of the old name. Both rebase clean onto each other. Assert the SECOND to land fails `verify` on the rebased tip and routes to needs-attention (or, with the host tier, is blocked from merging) — i.e. the clean rebase is NOT trusted. This is the test that proves the whole thesis; prior art: the existing `integration-core` fresh-worktree-gate tests + `advance-ci-template.test.ts`.
- **Concurrency tests at two levels.** In-process: two same-repo merge jobs through `run` with the `integrateLock` wired — one lands, the loser re-rebases + re-gates + lands or routes; never both-land-broken. Cross-job: simulate separate processes racing the same arbiter ref (the CAS loop alone) — assert deterministic convergence within the (resolved) cap, and the chosen queue/bounce behaviour past it.
- **Test external behaviour, not internals:** assert on what lands on `main`, what routes to needs-attention, and that `verify` ran on the rebased tree — not on which private function was called.

## Applied answers 2026-06-22

### q1: Cross-job merge serialiser for parallel merge in CI — git-alone floor + optional accelerator?

**Confirmed: (a) scaled `mergeRetries` as the git-alone FLOOR + (b) a ref-based cross-job land-lock as the portable ACCELERATOR. (c) GitHub `concurrency:` is allowed only as a host-specific convenience ON TOP, never as the serialiser the floor depends on.**

The brief's own framing forces this: the git-alone floor must be safe with nothing but `git push` + ref CAS against a bare arbiter, so the SERIALISER that guarantees correctness cannot be a GitHub-Actions feature. That rules (c) OUT as the floor. (a) is already the de-facto floor (the CAS loop IS the cross-job queue per story #5), so the only real decision is whether to ADD (b), and the answer is yes-but-carefully:

- **(a) is correctness-sufficient but throughput-poor under a burst.** `DEFAULT_MERGE_RETRIES = 5` was sized for in-process siblings. Past the cap a loser bounces to needs-attention — a SPURIOUS bounce (the work is fine; it just lost the race too many times). So (a) alone trades a real-but-bounded correctness floor for noisy false-positive attention under wide matrices.
- **Scale the cap, do not just leave it at 5.** Make `mergeRetries` resolve through the same precedence chain as the gate family (flag > env > per-repo > global > default) so a wide-matrix CI can raise it; keep the default modest. Crucially: a lost CAS must cost only a re-rebase + re-gate RETRY, never a `--force` and never a both-land-broken — which the engine already guarantees. Scaling the cap only changes WHEN a genuinely-stuck loser gives up, not the safety.
- **(b) the accelerator: a ref-based land-lock (e.g. CAS-claim a `refs/dorfl/land-lock` sentinel ref) so losers QUEUE/back-off rather than burn retries then bounce.** Portable (pure ref CAS, works against a bare arbiter), so it does NOT violate the floor framing the way (c) would. This is the cross-job analogue of the in-process `integrateLock`. Prefer it over (c) precisely so the accelerator degrades to every host, with GitHub merely faster, not required.
- **Guard against a stale land-lock.** A ref-lock held by a crashed job must be reclaimable (a TTL / staleness check, mirroring how the per-item lock reaper treats stale locks) or it becomes a self-inflicted deadlock that is strictly worse than (a)'s bounce. If a robust stale-lock story is not cheap, ship (a) scaled NOW and split (b) into a follow-on slice rather than ship a deadlock-prone lock.

Net: **(a) scaled = the shipped floor; (b) ref-lock = the portable accelerator, gated on a sound stale-lock reclaim; (c) = optional host sugar only.**

### q2: Runner-merges-propose: should the runner gain an explicit 'runner performs the propose merge' capability, or is the floor honestly 'push-time gated only, document the limitation'?
