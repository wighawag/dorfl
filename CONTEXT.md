# CONTEXT â€” agent-runner domain language

The domain glossary for `agent-runner`. Agents and skills should use THIS vocabulary when naming modules, tests, and discussing the system â€” it is the shared language. Architectural rationale lives in `docs/adr/` (decisions); product framing lives in `work/briefs/`. The authoritative contract is `work/protocol/WORK-CONTRACT.md`.

## What agent-runner is

A small TS/Node CLI that discovers, schedules, and runs work across many repos â€” both as a guided **human loop** and as an unattended **autonomous runner** â€” on top of a file-based `work/` contract and an atomic claim protocol. It is built by dogfooding itself (it tracks its own work in its own `work/`).

## Core domain terms

- **work/ contract** â€” the on-disk system this consumes (defined in `work/protocol/WORK-CONTRACT.md`, owned by the `setup` skill): one markdown file per item, **status = the folder it lives in** (never a field). Three regime umbrellas â€” `notes/` (capture buckets), `tasks/` (the build board), `briefs/` (the brief lifecycle) â€” plus top-level `questions/` and `protocol/`.
- **task** â€” one buildable work item: a tracer-bullet vertical task, a markdown file `work/tasks/todo/<slug>.md` (born staged in `work/tasks/backlog/`). Has frontmatter: `slug`, `brief`, `issue` (optional â€” only on a LONE `intake`-emitted task; the provider-agnostic closure link, mutually exclusive with `brief:`), `humanOnly` (usually omitted), `needsAnswers` (usually omitted), `blockedBy`, `covers`. (All field names are camelCase â€” see WORK-CONTRACT.md.)
- **brief** â€” a north-star doc in `work/briefs/ready/<slug>.md` a task's `brief:` field points at. (The launch/framing doc; may be a hand-off snapshot.) Frontmatter: `slug`, `issue` (optional), `humanOnly`/`needsAnswers` (the tasking gate), `briefAfter` (brief tasking-order). Tasked-ness is RESIDENCE in `work/briefs/tasked/` â€” there is no `tasked:` frontmatter marker (it was removed in `remove-sliced-marker-step-b`); see _brief lifecycle_ below.
- **ADR** â€” a decision record in `docs/adr/` (the _why_ of OUR technical choices; durable). Format = the **standard** one, owned at `work/protocol/ADR-FORMAT.md`: `NNNN-slug.md`, one decision per file, body = 1â€“3 sentences (context/decision/why), optional `status:` (`proposed | accepted | deprecated | superseded by ADR-NNNN` â€” most ADRs omit it). (Earlier slug-named/sectioned/`proposed`-lifecycle deviations were dropped; see `docs/adr/methodology-and-skills.md` Â§5/Â§5a. The two existing multi-decision files, e.g. `execution-substrate-decisions.md`, are grandfathered.)
- **capture buckets** â€” the THREE note-types under `work/` that are NOT work items and are **exempt from status=folder** (they don't flow/move; the folder is the inbox; they leave only by deletion): **`ideas/`** (proposed, pre-brief; _editable_), **`observations/`** (spotted, unverified signals; _append-only_), **`findings/`** (_verified external/domain_ ground truth â€” e.g. a reverse-engineered protocol; durable). Distinct from an ADR (what WE decided) and from an observation (internal â€śspotted, unverifiedâ€ť â€” NOT a finding). See WORK-CONTRACT.md.
- **slug** â€” content-derived, URL-safe id of an item (never a counter).
- **status (lifecycle)**: the folder: `backlog/` (claimable) â†’ `done/` (completed), or â†’ `dropped/` (the generic "won't-proceed" terminal, i.e. superseded / out-of-scope / duplicate / abandoned, with the REASON in the item body as `reason:`; generalises the previous `out-of-scope/`). Those terminal transitions are `git mv`. The TRANSIENT in-flight states are NO LONGER folders: `claimed` (was `in-progress/`) and `stuck` (was `needs-attention/`) are now a per-item lock ref (`refs/agent-runner/lock/<entry>`, `state: active`/`stuck`), not a `git mv` (see `claim (CAS)` and `needs-attention`). The body stays in `backlog/` while claimed.
- **autonomy gate (two axes)** â€” TWO orthogonal binary fields on tasks AND briefs (both default omitted = false): **`humanOnly`** (the DECIDED axis â€” _a human must drive this regardless of spec completeness_) and **`needsAnswers`** (the DISCOVERED axis â€” _unresolved questions block autonomous progress; the questions live in the body_). Orthogonal to lifecycle status. On a brief they gate _tasking_; on a task they gate _building_. (Supersedes the single-`humanOnly` gate, which replaced the three-state `afk` field.)
- **autoBuild policy** (`autoBuild`, per-repo) â€” _may agents auto-build undeclared items in this repo?_ Default `false`; resolved like `integration`: flag (`--auto-build`/`--no-auto-build`) > env (`AGENT_RUNNER_AUTO_BUILD`) > per-repo > global > default. The build member of the per-action gate family (`autoBuild`/`autoTask` for build/tasking; the question-surfacing gates `observationTriage`/`surfaceBlockers`). (`autoTask` is the auto-tasking gate's config key.) (No `allowAgents` key/flag/env alias exists, since this repo has no external users owed a migration window. See ADR `ci-config-policy-and-gate-family` and task `remove-deprecated-config-aliases`.)
- **observationTriage gate** (`observationTriage`, per-repo) â€” the 3-state `off | ask | auto` gate over the observation INBOX (raw captured signal). Default `off`; resolved like `integration`: flag (`--observation-triage <off|ask|auto>`) > env (`AGENT_RUNNER_OBSERVATION_TRIAGE`) > per-repo > global > default. `off` drops the observation pool from the advance auto-pick SELECTION (observations untouched); `ask` surfaces a promote/keep/delete question for each untriaged observation; `auto` auto-disposes the no-question cases (duplicate â‡’ recommend delete / unambiguous map) and asks about the rest (never auto-deletes a non-duplicate / auto-promotes). REPLACES the old `autoTriage` boolean (clean, no alias). An explicit `advance obs:<slug>` bypasses the selection gate and runs in `ask`-mode. The observation-side sibling of `surfaceBlockers` (which gates DECLARED blocked work); the two are orthogonal peers. See ADR `ci-config-policy-and-gate-family`.
- **blockedBy / eligibility** â€” an item is **agent-claimable** iff `needsAnswers` is not `true` AND `humanOnly` is not `true` AND `autoBuild` is `true`; it is **eligible now** iff also every `blockedBy` slug is present in the SAME repo's `work/tasks/done/`. Deps never cross repos.
- **brief lifecycle (folders)** â€” a brief flows through a folder state machine: **`work/briefs/proposed/`** (staging, pre-promotion) â†’ **`work/briefs/ready/`** (the auto-task pool) â†’ **`work/briefs/tasked/`** (tasked, resting), plus **`work/briefs/dropped/`** (the won't-proceed terminal). The held tasking LOCK (NOT a folder) marks a brief being tasked right now. The FOLDER is the SOLE source of truth for tasked-ness, exactly as `tasks/done/` is for tasks (there is no `tasked:` frontmatter marker â€” it was removed in `remove-sliced-marker-step-b`; the release transition records tasked-ness purely by moving the brief into `work/briefs/tasked/`). Re-task = `work/briefs/tasked/ â†’ work/briefs/ready/` (reopen-to-ready, mirroring `tasks/done/ â†’ tasks/todo/`).
- **briefAfter** (`briefAfter`, brief-only) â€” brief slugs that must already be **tasked** (resolved against `work/briefs/tasked/` residence, mirroring `blockedBy` â†’ `tasks/done/`) before the auto-tasker may task this brief â€” so its emitted tasks can reference the real slugs of those briefs' tasks in `blockedBy`. Distinct verb/signal from task `blockedBy` (which gates _building_, against `tasks/done/`).
- **needs-attention**: the post-claim **stuck** state, now a per-item LOCK state (`state: stuck` on `refs/agent-runner/lock/<entry>`), NOT a `work/needs-attention/` folder. It marks a claimed item that couldn't finish (red gate, conflict, ambiguity, timeout, rejected review). The runner AMENDS the held lock `active â†’ stuck` with the reason (plus any agent-surfaced questions) in the lock-entry body; there is NO `git mv` and NO on-`main` surface (so a protected-`main` bounce succeeds, and a work branch cut from `main` inherits no stuck record). A human resolves it via `resume`/`requeue`/`release-lock`. The human-visible surface is now `agent-runner status`/`scan` reading the lock refs (a command you run), not `ls` of a folder. (ADR `ledger-status-on-per-item-lock-refs`; brief `ledger-status-per-item-lock-refs`; ADR Â§12 for the original folder model.)
- **failure cause** â€” the CAUSE axis recorded on a needs-attention route + the `do` outcome / `run` `ItemStatus`, so a stuck item is not an undifferentiated "agent failed" (the cause drives the RECOVERY). It REUSES the existing terminal outcome vocabulary where it already fits and adds ONLY the two causes that vocabulary lacked â€” there is NO parallel naming scheme, and no new name duplicates an existing outcome (e.g. a red gate is the existing **`gate-failed`**, NOT a new `gate-red`):
  - **`gate-failed`** â€” the acceptance gate (`verify`) caught a genuine bug (fix the CODE). _(existing.)_
  - **`rebase-conflict`** â€” the rebase onto `<arbiter>/main` conflicted, aborted (resolve against main). _(existing.)_
  - **`review-blocked`** â€” Gate-2 (PR/code review) returned `block`. _(existing.)_
  - **`agent-stopped`** â€” the agent DELIBERATELY stopped on a drifted/ambiguous task, or produced no change (re-scope/re-claim). _(existing.)_
  - **`agent-failed`** â€” the agent RAN but produced bad/empty output, OR the cause is UNKNOWN. The CONSERVATIVE GENERIC + safe default: an unrecognised cause stays here (the classifier never forces a wrong specific label). _(existing name, reused as the default.)_
  - **`transient-infra`** _(NEW)_ â€” a harness-surfaced model/connection outage the harness surfaced AFTER its own retries, or a git/provider outage. RETRY the SAME work (the work is fine). Model-endpoint retries are the HARNESS's job (pi retries its own API); the runner only CLASSIFIES what is surfaced post-retry â€” it does NOT add model retries.
  - **`config-error`** _(NEW)_ â€” a thrown CORE wiring/config error (e.g. `review` on with no `reviewGate` wired). Fix the WIRING, not the task. The classification is BEST-EFFORT (lexical, from the surfaced error/detail) and is performed by the SHARED `classifyFailureCause` (`failure-cause.ts`) at BOTH the `do` and `run` failure-routing sites, so **`do` and `run` classify the SAME error the SAME way** (closing the old divergence where a thrown core error read as `usage-error` in `do` but `agent-failed` in `run`). (Task `failure-cause-classification-model-vs-git-vs-agent`.)
  - **`prepare-failed`** _(NEW)_ : the env-prep step (`prepare`, the install/codegen sibling of `verify`) was red, so the env could not be made ready and `verify` cannot be trusted. NEVER proceeds to `verify`/integrate. A distinct cause from `gate-failed` (the code is not necessarily wrong; the ENV is not ready). Fix the prepare step / the env, not the task. (Task `prepare-config-step`.)

## Claim & integration terms

- **arbiter** â€” the single git remote whose `main` ref serialises claims (GitHub, or a local `--bare` repo). Default remote name `origin`.
- **claim (CAS)**: atomically ACQUIRING an item's per-item lock (`refs/agent-runner/lock/<entry>`, `action: implement`) via a create-only `--force-with-lease` push; the first acquire wins, losers get exit 2 and pick another item. Implemented by `agent-runner claim` (`claim-cas.ts`). The claim writes NOTHING to `main`: the body STAYS at `work/tasks/todo/<slug>.md`, and the claimable predicate is "in `tasks/todo/` on `main` AND no lock held" (the held lock IS the claim). Because claim touches no protected ref, a protected-`main` repo is claimable. This SUPERSEDES the old direct-`main` mechanism (a `git mv` into an `in-progress/` folder plus a `main` claim commit), which is gone (brief `ledger-status-per-item-lock-refs`; ADR `ledger-status-on-per-item-lock-refs`). The ledger-transition seam (`docs/adr/claim-ledger-vs-protected-main.md`) still frames the transitions (claim / complete / needs-attention) so a future strategy could differ; the current strategy publishes claim/needs-attention on per-item lock refs and complete's durable move on `main`.
- **work branch** â€” `work/<slug>`, branched off the latest arbiter `main`, where a task is built.
- **integration mode** â€” how finished work lands: **`propose`** (push a branch + request review; the default) or **`merge`** (direct to main, opt-in for trusted/low-risk repos). Resolved at integrate-time: flag > per-repo > global > `propose`. Never `--force` to main. (ADR Â§6, Â§11.)
- **`taskingIntegration`** (per-TRANSITION TASKING override) â€” an OPTIONAL per-repo key (`propose`/`merge`) that overrides the integration mode for the **briefâ†’tasks (TASKING) transition ONLY** (a `do brief:<slug>` run: emit `work/tasks/backlog/*.md` + the `work/briefs/ready/ â†’ work/briefs/tasked/` lifecycle move). The flat `integration` keeps governing the **task-BUILD** transition (and stays the intake default). UNSET â‡’ tasking falls back to `integration`; the split happens at the option-threading caller, which threads **`taskingIntegration ?? integration`** into the tasking transition and plain `integration` into the build. The maintainer's target `integration:"propose"` + `taskingIntegration:"merge"` tasks a brief straight onto `main` (task FILES land, no PR) but builds each task as a reviewable PR. An explicit `--merge`/`--propose` ALWAYS wins for the transition the command runs (it resolves into BOTH keys upstream). DISTINCT from intake's per-EMITTED-TYPE `{task, brief}` (front-door, author-trust-resolved): `taskingIntegration` is per-LIFECYCLE-TRANSITION, inside the trust boundary, operator/config-resolved â€” two resolvers for two contexts, NOT to be re-forked into one. (Task `per-transition-integration-mode-slicing-vs-build`.)
- **`origin` / `originTrust`** (ORIGIN-TRUST PROVENANCE) â€” OPTIONAL frontmatter on a brief/task recording how it was BORN and the author-trust verdict at birth, so the trust signal SURVIVES the brief/task merge boundary. **`origin`**: `human` (default/unset â€” a local author, incl. a LOCAL `agent-runner intake` with no `--origin-trust`) or `issue` (the CI intake front-door stamped it). **`originTrust`**: `trusted` / `untrusted` (only meaningful when `origin: issue`); UNSET â‡’ `trusted` (the normal path, ZERO behaviour change). The chain: the CI `intake.yml` shell knows `author_association` and passes `--origin-trust <trusted|untrusted>` to `intake` (derived from the SAME `author_association` case as the integration flags, so they cannot desync); `intake` STAMPS `origin: issue` + that value (it does NOT resolve trust itself â€” that boundary stays in CI's policy); the tasker PROPAGATES the stamp onto every emitted task; and the task-**BUILD** transition resolves to `propose` for an `originTrust: untrusted` task even when the build `integration` is `merge` (the risk is the becomes-code build; a brief/task FILE on main is inert). Precedence on the build transition: **explicit `--merge`** (the operator is present; CLI always wins, no special force-key) > **untrusted-origin â‡’ propose** > `integration` (config) > default. The autonomous/CI build path passes no flag, so untrusted-origin reliably forces propose. DISTINCT from intake's per-EMITTED-TYPE integration (that decides the emit-time mode of the artifact; this is PROVENANCE that survives the merge so the build-time checkpoint is not laundered) and from `taskingIntegration` (a per-transition mode override â€” this is a per-artifact provenance stamp the build transition READS). (Task `untrusted-origin-forces-build-propose`.)
- **verify (the gate)** â€” the per-repo acceptance command (`agent-runner verify`, e.g. `pnpm -r build && test && format:check`). The deterministic trust boundary: authoritative & non-skippable for the autonomous runner; default-on but `--skip-verify` for the human `complete`. (ADR Â§8.)

## Execution-substrate terms

- **job** â€” one claimed item being processed (there is NO long-lived "agent" identity; the unit is the job). (ADR Â§1.)
- **hub mirror** â€” one bare mirror per repo under `~/.agent-runner/repos/<key>.git` (the shared `repo-mirror` primitive); cheap shared object store.
- **worktree** â€” a job's isolated working tree off the hub mirror, under `~/.agent-runner/work/<work-id>/`, on branch `work/<slug>`. (ADR Â§2.)
- **human worktree** (`work-on`) â€” the HUMAN counterpart of a job worktree: an isolated worktree off the hub mirror on `work/<slug>`, but checked out under a **human-friendly** root (`humanWorktreesDir`, e.g. `~/worktrees/<key>/<slug>/`) so a person can edit several tasks in parallel. Deliberately NOT under `~/.agent-runner/` (the agents' area) â€” so it never carries a human's secrets into an agent context. Two forms (`work-on <slug>` in-repo, `work-on <remote> <slug>` anywhere): the ONLY difference is the worktree's LOCATION â€” both claim the slug and branch off the freshly-fetched `<arbiter>/main` (same claim, same starting commit). `--copy <patterns>` copies named gitignored files (copy, not symlink; `--copy-from` required in the remote form) with a security notice. A binary can't `cd` your shell, so it prints the path + a `cd` hint; `--print-dir` emits the path only, for a shell wrapper: `work-on(){ cd "$(agent-runner work-on "$@" --print-dir)"; }`.
- **humanWorktreesDir** â€” config root for human worktrees, prompted + saved on first use (sensible suggestion, no silent default; chosen to NOT share a prefix with code dirs so shell tab-completion never collides). Never `~/.agent-runner/`.
- **work-id** â€” flat, deterministic key for a job: `<host-...>__<org>__<name>__<slug>` (the repo key with `.`â†’`-`, then the slug).
- **repo key** â€” hierarchical `host/org/name` with `.`â†’`-` per segment (`github-com/wighawag/agent-runner`).
- **deletion predicate** â€” a job worktree is removed only when its work is **provably on the arbiter** (clean tree AND branch tip reachable on the arbiter); otherwise retained (a retained worktree is a needs-attention signal). `gc` re-applies it. (ADR Â§4.)
- **harness seam** â€” pluggable interface for launching a job's agent and reporting liveness. Adapters: `null` (shells out to `agentCmd`, PID-only liveness) and **`pi`** (invokes the pi CLI with the work-agent prompt; liveness from PID + a pointer to the pi session dir/log). Selected via the `harness` config; jobs' liveness is resolved per-record to the owning adapter (`resolveHarness`). Liveness from the harness, **never filesystem mtime**. (ADR Â§5.)
- **integration seam** â€” `Integrator` with two orthogonal axes: **mode** (`merge`/`propose`) and **PR-INTENT** (`noPR` + `--no-pr`: suppress the PR even on an authed GitHub arbiter). The **provider** (`github` via `gh`, `none` = push + open-manually) is **purely ARBITER-DERIVED** (a GitHub arbiter URL â‡’ `github`, else `none`) â€” there is NO `provider` override/`--provider` flag (a stale `provider` key warns + is ignored; the old `provider: none` use is re-homed to `noPR`). Whether `gh` can open the PR is the identity's `providers.github` / ambient `gh` auth. A `propose` run that INTENDS a PR (`noPR` unset) on a GitHub arbiter where a `gh` auth PROBE says `gh` can't open one FAILS UP FRONT (pre-flight), never a silent degrade â€” but absent-identity-with-ambient-`gh`-auth still proceeds. Push is the safety-bearing action. (ADR Â§6.)

## The faces (commands) â€” see `docs/adr/command-surface-and-journeys.md`

Organised by two axes: **target** (the registry / one repo) Ă— **doer** (agent / human). The full model + rationale is the ADR; this is the glossary view.

- **Registry (what gets watched):** **`remote add <url> [--local]`** (register a target â†’ create its hub mirror; `--local` = a `--bare` arbiter), `remote rm`, `remote ls`, **`remote find <folder>`** (discover `work/`-participating repos, toggle-add). The registered set IS the set of hub mirrors â€” there is no `roots` or `remotes` config field. Key = `host/org/name`; `remote add` guards against registering one project under two transports.
- **Autonomous â€” agent does it:**
  - **`run`** â€” the cross-repo, **parallel** daemon: scan the registry, claim up to `maxParallel`, run agents concurrently in job worktrees, integrate, loop forever (future service). **`run --once`** = one tick (debug/test the daemon; NOT the CI path).
  - **`do`** â€” the per-repo, in-place **worker**: claim+build+integrate in ONE repo, then exit. `do <slug>` / `do brief:<slug>` (task it) / `do` (auto-pick) / `do <slug>â€¦` / `do -n <x>`; `--propose` (default) / `--merge`. **This is the CI command.** In a checkout it works in-place; `do --remote <r>` materialises a hub mirror + job worktree in the agents' area.
- **Human â€” do work yourself (optionally with your AI):**
  - **In-place** (takes over the current checkout, for when you need its real `.env`): **`start <slug>`** (claim + switch; `--agent` launches the harness) â†’ build â†’ **`complete`** (gate + done-move + integrate). **`resume <slug>`** re-engages an in-progress item. `claim`/`prompt` are low-level.
  - **Parallel** (isolated worktree, doesn't touch your clone): **`work-on <slug>` / `work-on --remote <r> <slug>`** â€” claim + worktree in the **human area**, `cd`s you in; `--copy` brings gitignored files (e.g. `.env`), `--agent` launches the harness. The human counterpart to `do`.
- **Ops:** **`scan`** (cross-repo queue â€” fetches the truth, warns+falls back offline), **`status`** (running/stuck/cleanup dashboard), **`requeue <slug>`** (give up a stuck hold and return the item to the pool by releasing its lock; the defer-don't-finish verb), **`gc`** (reap job worktrees, never mirrors), **`verify`** (run the gate standalone).

## Invariants (do not relitigate â€” see ADRs)

- The **runner owns all git-state transitions**; the build agent only writes code and gets the gate green (stated in-band in the agent prompt).
- **Status = folder, never a field** (conflict-safety). One file per item; no shared index; content slugs, never counters.
- Conflicts: **rebase-or-abort, never auto-resolve** â†’ needs-attention. (ADR Â§10.)
- **Storage areas map onto the doer axis:** agent execution â†’ agents' area (`~/.agent-runner/`, hub mirrors + job worktrees); a human doing the work â†’ human area (`humanWorktreesDir`, never under `~/.agent-runner/`). The secrets-isolation boundary. (`command-surface-and-journeys` Â§2.)
- **adopt = skill, execute = command:** adopting the contract (`setup` â€” the single onboarding/migration skill â€” plus the `to-brief`/`to-task` methodology) is protocol-layer (a SKILL, runner-agnostic); executing work (claim, `run`, `do`, integration) is implementation-layer (a command). Reinforces ADR Â§9. (`command-surface-and-journeys` Â§8.)
- **Skills live IN THIS REPO under `skills/`; `~/.agents/skills/*` are SYMLINKS into them.** The agent-runner skills (`setup`, `to-brief`/`to-task`, `drive-tasks`, `orchestrate`, `review`, `surface-questions`, `triage-observations`, `capture-signal`) are authored here and the user's `~/.agents/skills/<name>` entries point at `skills/<name>` by symlink. Consequences: editing `skills/` IS editing the live skill (no copy/propagate step, unlike the `work/protocol/` mirror); a skill RENAME is in-repo `git mv` work that belongs in the relevant task; and a renamed skill directory needs its `~/.agents/skills/` symlink re-pointed by the user (outside this repo). Do not treat the skills as external/read-only. (The `folder-taxonomy-reorg-and-rename` rename moved `skills/to-prd -> skills/to-brief` and `skills/to-slices -> skills/to-task`; their `~/.agents/skills/to-prd` and `~/.agents/skills/to-slices` symlinks must be re-pointed to the new dirs by the user. Those slug strings are the real historical directory names, kept verbatim.)
- **`scan`'s offline guarantee is RETIRED:** in the registry model the remote is the source of truth, so `scan`/`status` fetch-first (warn + fall back offline). (Supersedes the older roots-local "scan is always offline" framing.)
- **Agents NEVER modify workflow files (`.github/workflows/*`).** The CI identity is a GitHub App token without `workflows` permission, so a push that touches a workflow file is `[remote rejected]` and the branch strands runner-local + unrecoverable. So a build task must leave `.github/workflows/*` untouched; CI workflows are regenerated from their seed/template (`docs/ci/*.template` + the `*-template.ts` emitter) by a HUMAN running `install-ci` later. Edit the template/emitter (the source), never the emitted workflow.

## Coherence (a first-class quality)

**Consistency and conceptual coherence are a stated quality of this project**, not a nicety. THIS glossary + the ADRs are the single source of truth for what each term means. A new flag / config key / status / verb / named concept MUST NOT silently re-mean an existing term, mean two different things in two places, sit at the wrong conceptual layer, or duplicate an existing concept under a new name. Before adding a concept, check it against this glossary + the ADRs + the code; if it conflicts, re-means, or overlaps, REUSE or RENAME rather than fork (or surface it). This is enforced by the `review` skill's **conceptual-coherence lens** and the Gate-2 review prompt, and prevented up-front by the CLAIM-PROTOCOL coherence check. Rationale: a muddled concept that compiles is far more expensive than the question â€” every later artifact that reuses the muddled term inherits the debt (the `autoTask`-gated-the-VERB-not-the-SELECTION miss is the worked example; `work/notes/findings/autoslice-gate-conflates-verb-autonomy-and-review-loop.md`).

## House style

pnpm monorepo (CLI in `packages/agent-runner/`), `type: module`, NodeNext, tsc, prettier (tabs + single quotes), vitest, `commander`. Tests use throwaway git repos + a local `--bare` arbiter. Acceptance gate is `pnpm -r build && pnpm -r test && pnpm -r format:check` (see `AGENTS.md`).
